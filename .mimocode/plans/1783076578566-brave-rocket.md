## 二、改进方案

### 模块 A：智能需求分解器（解决 G1）

**目标**：用户输入万字以上长提示词时，由协调者 Agent 自动拆解为结构化研究计划。

**新增文件**：
- `backend/app/agents/requirement_decomposer.py` — 需求分解 Agent

**修改文件**：
- `backend/app/routers/tasks.py` — 提交时判断长度，>3000 字触发分解
- `backend/app/core/task_persistence.py` — 存储分解结果到 `requirement_plan.json`
- `backend/app/agents/langgraph_orchestrator.py` — preflight 节点前插入分解步骤

**设计细节**：
```
用户输入 → 判断长度(>3000字) → RequirementDecomposer Agent
  ↓
输出结构化 JSON：
{
  "research_goal": "一句话研究目标",
  "background": "研究背景摘要",
  "key_questions": ["Q1", "Q2", ...],
  "methodology_hints": ["方法提示1", ...],
  "expected_output": "预期产出类型",
  "data_requirements": ["需要的数据类型"],
  "template_suggestion": "推荐模板ID",
  "subtasks": [
    {"id": 1, "description": "...", "agent": "research_agent"},
    {"id": 2, "description": "...", "agent": "modeler_agent"},
    ...
  ]
}
  ↓
存储到 project 目录/requirement_plan.json
所有 Agent 通过 context 读取此文件
```

**存储位置**：`{project_dir}/tasks/{task_id}/requirement_plan.json`

### 模块 B：多 Agent 协作讨论与投票系统（解决 G2 + G7）

**目标**：Agent 间可发起结构化讨论，通过投票达成共识；人类可随时加入。

**新增文件**：
- `backend/app/core/agent_discussion.py` — 讨论协议与投票引擎
- `backend/app/routers/discussion.py` — 讨论 API 路由

**修改文件**：
- `backend/app/agents/langgraph_orchestrator.py` — 在关键决策点插入讨论节点
- `backend/app/core/memory.py` — WorkingMemory 增加 discussion 分区

**设计细节**：

```
讨论协议 (DiscussionProtocol):
  - proposal: Agent 提出方案（含 rationale）
  - discussion_round: 其他 Agent 逐个发表意见（支持/反对/修正）
  - vote: 每个 Agent 投票（approve/reject/abstain）
  - decision: 多数决 or 达成共识
  - human_override: 用户可随时介入并拍板

触发时机（关键决策点）：
  1. 研究方法选择（research_agent vs modeler_agent 方案对比）
  2. 模型选择（多个候选模型的对比讨论）
  3. 实验设计（baseline/metric/ablation 的方案确认）
  4. 论文结构（章节安排的最终确认）

投票规则：
  - 简单多数（>50% approve）
  - 人类用户一票否决权
  - 讨论轮次上限：3 轮（防止死循环）
```

**人类介入机制**：
- 前端 DiscussionPanel 实时展示讨论进展
- 用户可发送消息参与讨论（通过 chat_room）
- 用户可直接拍板覆盖 Agent 决策
- 讨论结果存入 WorkingMemory 的 `decisions` 分区

### 模块 C：任务完成总结与经验提取（解决 G3 + G8）

**目标**：任务结束后自动生成结构化总结报告，并提取经验存入记忆池。

**新增文件**：
- `backend/app/agents/summary_agent.py` — 任务总结 Agent

**修改文件**：
- `backend/app/agents/langgraph_orchestrator.py` — 工作流末端插入总结节点
- `backend/app/core/agent_memory.py` — 增强经验提取逻辑
- `backend/app/core/memory.py` — LessonsMemory 增加分类维度

**设计细节**：

```
总结报告结构 (TaskSummary):
  - research_summary: 研究过程回顾（用了什么方法、找了什么文献）
  - innovation_points: 创新点总结
  - experiment_results: 实验结果摘要
  - paper_quality: 论文质量自评（各章节评分）
  - lessons_learned: 经验教训（分为方法选择/写作/数据处理/实验设计 四类）
  - dataset_registry: 使用的数据集清单
  - literature_registry: 引用的文献清单
  - recommendations: 对后续工作的建议

经验分类体系:
  category: method_selection | writing | data_processing | experiment_design | template_specific
  subcategory: 根据模板细化（如 CCF-A 写作 / 竞赛写作）
  tags: [具体方法名, 工具名, 数据类型]
  impact_score: 1-10（基于任务完成质量）
  usage_count: 检索使用次数
```

**存储**：
- 总结报告：`{project_dir}/tasks/{task_id}/task_summary.json`
- 经验条目：`data/memory/lessons.json`（现有格式扩展）
- Agent 案例：各 Agent 的 `{agent_name}_cases.jsonl`

### 模块 D：知识库自动分类整理（解决 G4）

**目标**：下载的数据集、文献自动归类到全局知识库。

**新增文件**：
- `backend/app/services/knowledge_organizer.py` — 知识整理服务

**修改文件**：
- `backend/app/agents/langgraph_orchestrator.py` — 任务结束后触发整理
- `backend/app/core/knowledge_manager.py` — 增加分类标签 API

**设计细节**：

```
自动分类维度:
  按资源类型: paper | dataset | code | benchmark | survey
  按学科领域: NLP | CV | RL | optimization | time_series | ...
  按方法类型: transformer | CNN | GNN | reinforcement_learning | ...
  按质量等级: top_venue | good | average | low

整理流程:
  1. 任务完成后扫描 {task_dir}/downloads/
  2. 对每个文件进行分类（基于文件名+内容摘要）
  3. 移动到全局知识库对应分类目录
  4. 注册到 knowledge_manager 的 KB 索引
  5. 生成分类目录索引

存储结构:
  data/knowledge_bases/
    global/
      papers/
        {year}/{venue}/{paper_id}.pdf
      datasets/
        {domain}/{dataset_name}/
      code/
        {method_name}/
    index.json  # 分类索引
```

### 模块 F：实验闭环改进（解决 G6）

**目标**：experimentation_agent 输出实验方案后，可根据结果自动迭代优化。

**修改文件**：
- `backend/app/agents/langgraph_orchestrator.py` — experimentation 后增加评估节点
- `backend/app/agents/base.py` — 增加实验结果评估 hook

**设计细节**：

```
实验闭环:
  experimentation_agent 设计实验
    ↓
  solver_agent / experimentation_agent 执行实验
    ↓
  结果评估节点（检查指标是否达标）
    ↓ 未达标
  experimentation_agent 调整方案（修改超参/baseline/metric）
    ↓
  重新执行（最多 3 轮迭代）
    ↓ 达标或达到上限
  进入写作阶段

评估标准:
  - 是否包含 baseline 对比
  - 是否有 ablation study
  - 指标是否与 SOTA 可比
  - 实验结果是否支撑论文论点
```

## 四、关键设计决策

### 决策 1：需求分解器的触发条件
- **方案 A**：固定阈值（>3000 字）— 简单可靠
- **方案 B**：LLM 判断复杂度 — 更智能但增加延迟
- **推荐**：方案 A（简单阈值），复杂度判断留给 preflight agent

### 决策 2：讨论投票的共识机制
- **方案 A**：简单多数决 — 快速收敛
- **方案 B**：加权投票（按 Agent 专业度） — 更准确
- **推荐**：方案 A + 人类一票否决权

### 决策 3：知识库分类的实现方式
- **方案 A**：基于文件名 + 规则匹配 — 快速但不精确
- **方案 B**：LLM 分类 — 精确但消耗 token
- **推荐**：方案 B（LLM 分类，使用轻量模型），结果缓存

### 决策 4：经验存储格式
- **方案 A**：扩展现有 lessons.json — 向后兼容
- **方案 B**：迁移到 SQLite — 更强查询能力
- **推荐**：方案 A（保持 JSON 格式，扩展字段），避免迁移成本
