## 一、系统定位与技术栈全景

### 1.1 系统定位

本系统是一个**面向 CCF-A 顶会 / 数学建模竞赛的多智能体协作论文辅助生成平台**。系统以自然语言问题描述 + 可选数据文件为输入，由 10+ 个专业化 Agent 组成的团队协作完成从问题分析到 Camera-Ready 论文打包的全流程，最终输出结构完整的 LaTeX 论文。

**关键定位：辅助工具，非替代研究者。** 系统自动化了论文写作中的重复性工作（格式排版、图表生成、文献整理、代码执行），但论文的核心学术价值（研究想法、方法创新、实验设计）仍取决于输入的研究方向和 LLM 的能力。

## 二、核心架构设计

### 2.1 LangGraph StateGraph 编排器

**设计动机**：原有 Orchestrator 是硬编码的 if-else 控制流（1500+ 行），新增节点需要修改多处条件分支，维护成本高。LangGraph 的声明式节点-边模型让工作流变更只需修改图结构，不触碰节点内部逻辑。

**状态定义**（`TaskState` TypedDict）：

```python
class TaskState(TypedDict, total=False):
    messages: List[Dict[str, Any]]      # 对话历史
    files: List[str]                     # 数据文件路径
    preflight: Optional[Dict[str, Any]] # Preflight 决策报告
    current_step: str                    # 当前执行步骤
    paper_template: str                  # 论文模板 ID
    workflow_type: str                   # standard/quick/deep_research/code_focused/research_paper
    mode: str                            # batch/sequential
    phase: str                           # 当前阶段
    retry_count: int                     # 重试计数
    escalation_count: int                # 升级计数
    solver_attempts: List[Dict]          # 求解尝试历史
    artifact_paths: List[str]            # 产物路径
    cannot_solve_report: Optional[Dict]  # 无法求解报告
    task_id: str
    problem_text: str
    project_name: Optional[str]
    knowledge_base_id: Optional[str]
    results: Dict[str, Any]              # 所有 Agent 输出汇总
    sub_problems: List[Dict[str, Any]]   # 子问题列表
    should_pause: bool                   # 是否暂停等待用户
    revision_count: int                  # 论文修订轮次
```

**节点图**（18 个节点 + 条件边）：

```
preflight_decision ──→ analyzer ──→ data ──→ research ──→ discuss_approach
       │                                                        │
       ├──→ self_collect ──→ preflight_decision (loop)          ▼
       │                                                  modeler ──→ experiment
       └──→ cannot_solve ←──────────────────────────────────────┘    │
              ↑                                                       ├──→ iterative_solver
              │                                                       │         │
              └─── cannot_solve ◄───────────────────────────────────────┘         ├──→ figure
                                                    ├─→ wait_user ──→ END       │
                                                    └─→ revise (loop) ◄───────────┤
                                                    peer_review ←── writer ←────┘
                                                           │
                                                           └──→ fact_check ──→ END
```

**条件边**：

| 源节点 | 路由函数 | 分支 |
|--------|---------|------|
| `preflight_decision` | `_route_preflight` | standard / quick / deep_research / code_focused / research_paper / self_collect / abort |
| `peer_review` | `_route_peer_review` | accept / revise / wait_user / abort |
| `iterative_solver` | `_route_solver` | success / retry / escalate / abort |
| `research` | `_route_after_research` | discuss / modeler / algorithm_engineer / financial_analyst / writer |
| `analyzer` | `_route_after_analyzer` | data / research / modeler / algorithm_engineer / financial_analyst / writer |
| `data` | `_route_after_data` | research / modeler / algorithm_engineer / financial_analyst / writer |
| `discuss_approach` | `_route_after_discuss_approach` | modeler / algorithm_engineer / financial_analyst / writer |
| `modeler` / `algorithm_engineer` / `financial_analyst` | `_route_to_experiment_or_solver` | experiment / iterative_solver |

**断点续传**：系统支持从 checkpoint 恢复任务。当任务被中断（系统重启、进程崩溃）后，重新启动任务时会自动从最后一个 checkpoint 恢复状态，继续执行而非从头开始。

**兼容层**：`orchestrator.py` 的 `execute_workflow` 检测 `use_langgraph_orchestrator=True` 后委托给 `LangGraphOrchestrator.run()`，旧路径保留不变，通过 `config.py` 的 `use_langgraph_orchestrator` 开关切换。

### 2.2 Preflight 决策引擎

**设计动机**：原有系统在 `submit_task` 时不做任何数据/流程判断，直接跑 phase1。导致两个问题：
1. 无数据时 LLM 凭空编造数值（幻觉）
2. 问题类型不匹配时走错工作流

**决策流程**：

```
用户提交任务
    │
    ▼
DataSchemaExtractor.extract()
    │  提取：行数/列数/列类型/数值范围/缺失率
    ▼
LLM 综合判断（单次调用，ReAct-style）
    │  输出：problem_type / has_data_confidence / data_subjects
    │        recommended_template / recommended_workflow / recommended_mode
    │        data_adequacy / llm_should_collect / collection_plan
    ▼
┌─ data_adequacy == "sufficient" ──→ 正常启动
├─ data_adequacy == "missing" + llm_should_collect ──→ self_collect（自主搜索下载数据）
├─ data_adequacy == "missing" + 无法搜集 ──→ 422 要求上传
└─ data_mismatch_warning ──→ 422 + 报告
```

**关键数据结构**（`PreflightReport`）：

```python
@dataclass
class PreflightReport:
    problem_type: str                    # 优化/预测/评价/分类/仿真/网络/物理/测量/综合
    has_data_confidence: float           # 0-1，数据与题目的匹配度
    data_subjects: List[str]             # 数据主题（金融时序/医学影像/工程测量/...）
    recommended_template: str            # 8 模板之一
    recommended_workflow: str            # standard/quick/deep_research/code_focused/research_paper
    recommended_mode: str                # batch/sequential
    data_adequacy: DataAdequacy          # sufficient/insufficient/missing
    llm_should_collect: bool             # 是否建议自主搜集
    collection_plan: str                 # 搜集计划（搜什么、去哪搜）
    data_mismatch_warning: Optional[str] # 数据不匹配警告
    data_schemas: List[Dict]             # 每份数据的 schema 摘要
```

### 2.3 ReAct 工具循环

**设计动机**：原有 Agent 的 `call_llm` 是单次调用，LLM 无法在推理过程中使用工具（搜索文献、读取数据、执行代码）。引入 ReAct 循环后，Agent 可以在 Thought-Action-Observation 周期中自主决策工具使用。

**实现架构**：

```python
async def call_llm(self, messages, tools=None, max_react_iterations=5, ...):
    if not tools:
        return await self._call_llm_once(messages, temperature)

    # ReAct 循环
    for iteration in range(max_react_iterations):
        response = await self._call_llm_once(messages, temperature, tools=tools_payload)
        tool_calls = self._parse_tool_calls(response, api_format)
        if not tool_calls:
            return response  # LLM 给出最终答案

        # 执行工具，拼回 messages
        for tc in tool_calls:
            observation = await self._execute_tool_call(tc)
            messages.append(self._build_tool_result_message(tc, observation, api_format))
```

**工具注册表**（5 种内置工具 + MCP 扩展）：

| 工具名 | 功能 | 实现 |
|--------|------|------|
| `search_web` | 文献搜索 | 接入 `research_agent.execute()` |
| `read_csv_columns` | 读取数据列信息 | 接入 `DataSchemaExtractor.extract()` |
| `run_python` | 沙箱执行 Python | `subprocess.run` + 资源限制 |
| `write_python_snippet` | 写文件 | `Path.write_text` |
| `read_paper` | 读取论文元数据 | 接入 Semantic Scholar + MetadataCache |
| `arxiv_search` | arXiv 论文搜索 | MCP 服务器 |
| `arxiv_download` | arXiv PDF 下载 | MCP 服务器 |
| `scholar_search` | Google Scholar 搜索 | MCP 服务器 |

**Provider 适配**：

| API 格式 | tools 格式 | tool_calls 解析 |
|----------|-----------|----------------|
| `anthropic` | `{"name", "description", "input_schema"}` | `content[].type == "tool_use"` |
| `openai_chat` | `{"type": "function", "function": {...}}` | `message.tool_calls[]` |
| `ollama_chat` | 同 OpenAI | 同 OpenAI |
| `gemini_native` | 暂不支持 | 忽略 tools，走普通调用 |

## 三、多智能体协作机制

### 3.1 Agent 团队

| Agent | 角色 | 核心能力 | 默认模型 |
|-------|------|---------|---------|
| `analyzer_agent` | 问题分析师 | 子问题分解、问题类型识别、难度评估 | sonnet |
| `data_agent` | 数据分析师 | 数据文件解析、schema 提取、洞察抽取 | sonnet |
| `research_agent` | 研究员 | arXiv 检索 + Semantic Scholar 增强 + Reranker | haiku |
| `modeler_agent` | 建模师 | 数学建模、模型选择、算法推荐 | sonnet |
| `solver_agent` | 求解器 | 代码生成 + 沙箱执行 + 自动修复 + 跨方法验证 | sonnet |
| `writer_agent` | 写作专家 | 按章节独立 LLM 调用、LaTeX 生成、自评 critique | opus |
| `peer_review_agent` | 审稿人 | 4 维评分（novelty/soundness/clarity/significance）+ 可复现性检查 | sonnet |
| `experimentation_agent` | 实验设计 | baselines/datasets/metrics/ablation plan | haiku |
| `coordinator` | 协调者 | 工作流编排、暂停/恢复、跨 Agent 黑板 | - |

**AgentModelRouter**：按 `(agent_name, template_id)` 路由模型。CCF-A 4 套模板强制 `writer = opus`。

### 3.2 多 Agent 讨论协议

**设计动机**：原有系统是线性数据流（A → B → C），Agent 之间没有真正的"讨论"。引入讨论节点后，多个 Agent 可以在关键决策点互相审阅、提出建议。

**`discuss_approach` 节点**：

```
分析师: "我认为应该用线性规划，因为..."
建模师: "同意，但需要考虑约束的非线性..."
研究员: "arXiv 上有一篇类似问题用混合整数规划..."
协调者: "综合各方意见，采用混合整数规划方案。"
```

**用户参与机制**：

- 前端 AgentChat 组件支持用户输入消息（`POST /{task_id}/messages`）
- 用户消息实时出现在 Agent 讨论流中（SSE 推送）
- `_route_peer_review`：用户已发言 → `wait_user`（等待用户指导）；用户未发言 → 自动修订（最多 3 轮）

### 3.3 自主迭代求解

**设计动机**：借鉴 Claude Code Plan mode 的"自主迭代改进直到求出答案"理念。solver 失败后不是立即停止，而是在 Harness 评判下自动修复，直到成功或集体判定无法求解。

**迭代流程**：

```
solver_agent.execute()
    │
    ▼
Harness 评判（ResultValidator + CrossValidator + CodeManifest）
    │
    ├── passed ──→ 进入下一子问题
    │
    └── failed ──→ _classify_execution_error
                    │  注入错误分类 + 修复建议到下一轮 prompt
                    ▼
              solver_agent.execute()（第 2 次）
                    │
                    ├── passed ──→ 进入下一子问题
                    │
                    └── failed ──→ ...（最多 max_solver_iterations 次）
                                    │
                                    ▼
                              _multi_agent_vote
                                    │
                    ┌───────────────┼───────────────┐
                    ▼               ▼               ▼
                  retry        collect_data       abort
                  (升级重试)    (搜集数据)       (无法求解)
```

**多 Agent 投票**（`_multi_agent_vote`）：

向 `analyzer_agent`、`modeler_agent`、`peer_review_agent` 发送诊断 prompt，每个 Agent 独立判断失败原因（代码 bug / 数据不足 / 无法继续），多数投票决定后续行为。

## 四、Harness 校验体系（5 层验证）

**设计动机**：论文中的数值必须有真实来源，不能靠 LLM 幻觉。5 层校验从不同维度确保结果的正确性。

| 层级 | 服务 | 文件 | 校验内容 |
|------|------|------|---------|
| L1 | ResultValidator | `services/result_validator.py` | NaN/Inf/负数/范围/一致性（准确率>1.0 为 error） |
| L2 | CrossValidator | 同上 | 5% 阈值跨方法比对 |
| L3 | CodeManifest | `services/code_manifest.py` | 300 行/多角色/多子问题拆分校验 |
| L4 | ContractValidator | `services/contract_validator.py` | Pydantic schema 校验 Agent 输出 |
| L5 | FactChecker | `services/fact_checker.py` | main.tex 数字与 solves.json 对账 |

**ResultValidator 问题类型**：

| level | type | 描述 |
|-------|------|------|
| error | `empty` | 结果为空 |
| error | `nan` | 含 NaN |
| error | `inf` | 含 Inf |
| error | `negative` | 非负字段为负 |
| error | `range` | R² > 1.0 / 准确率 > 1.0 / 精确率 < 0 |
| warning | `magnitude` | 数值过大/过小 |
| warning | `consistency` | AHP 的 CR ≥ 0.1 |

**CrossValidator**：跨方法比对，差异 > 5% 自动报警。支持注册 `ALTERNATIVE_METHODS`（`analytical_estimate` / `llm_second_opinion`）。

**FactChecker**：从 `main.tex` 提取所有数字，与 `solves.json` 比对，超阈值报警。确保论文中的每个数值都有真实来源。

## 五、GPU 实验执行与 OOM 防护

### 5.1 GPU 执行器架构

```
ExperimentRunner
├── _run_single_gpu()        # 优先路径：使用 GPUExecutor
│   ├── GPUExecutor.execute_training()
│   │   ├── VRAMMonitor.start()     # 启动显存监控线程
│   │   ├── subprocess.run()        # 执行训练脚本
│   │   ├── VRAMMonitor.stop()      # 停止监控
│   │   └── TrainingLogParser.parse() # 提取指标
│   └── CheckpointManager.save_checkpoint_meta()
│
└── _run_single_sandbox()    # 回退路径：使用 CodeSandbox（无 GPU）
```

### 5.2 显存监控与 OOM 防护

**VRAMMonitor**：独立监控线程，每 2 秒检查一次 GPU 显存使用。

| 阈值 | 行为 |
|------|------|
| 85% | 记录 warning 日志 |
| 95% | 触发 `torch.cuda.empty_cache()`，标记训练失败 |

**显存估算**：训练前自动估算推荐 batch size。

```python
gpu_executor.get_recommended_batch_size(
    model_memory_mb=500,      # 模型预估显存
    per_sample_memory_mb=10,  # 每样本预估显存
)
```

### 5.3 检查点保存与恢复

**CheckpointManager**：
- 自动保存检查点索引（`checkpoints.json`）
- 支持从最新检查点恢复训练
- 自动清理旧检查点（默认保留 3 个最新）

**训练脚本约定**：训练脚本需支持 `--resume_from` 参数和 `--output_dir` 参数，并在保存检查点时输出标准格式日志：

```python
print(f"CHECKPOINT Saved epoch={epoch} step={step} path={ckpt_path} metric=val_acc:{val_acc:.4f}")
```

**恢复流程**：
```
任务中断 → 重启任务 → _restore_from_checkpoint() → 加载最新 checkpoint → 从该 epoch 继续训练
```

## 六、多智能体记忆与幻觉抑制

### 6.1 三级记忆架构

灵感来自 **G-Memory: Tracing Hierarchical Memory for Multi-Agent Systems** (NeurIPS 2026)。系统采用三层记忆：

```
MemoryManager
├── Working Memory（任务级黑板）
│   ├── problem: {text, type, template, difficulty}
│   ├── sub_problems: List[Dict]
│   ├── data_insights: List[str]
│   ├── literature: List[Dict]
│   ├── methods: List[Dict]
│   ├── agent_results: Dict[str, Any]
│   └── decisions: List[Dict]
│
├── Episodic Memory（事件流）
│   └── events: List[{agent, action, detail, timestamp}]
│
└── Lessons Memory（跨任务经验）
    └── lessons: List[{problem_type, lesson, use_count, score}]
```

### 6.2 幻觉抑制体系

**文献引用验证**：
- arXiv ID 验证：通过 `arxiv.org/abs/{id}` HTTP HEAD 请求
- 验证状态：`verified=1.0` / `unverified=0.3` / `phantom=0.0`

**数值结果验证**：
- ResultValidator 检查数值范围（准确率不可能 > 1.0）
- CrossValidator 跨方法比对（差异 > 5% 报警）
- FactChecker 论文数字与求解结果对账

**Mock 结果标记**：
- 所有 mock 结果统一增加 `"_source": "mock"` 标记
- `ResultValidator` 跳过 mock 数值验证，UI 显示 "演示模式"

## 七、数据驱动的全流程决策

### 7.1 数据流链路

```
用户上传文件 → FileManager.tsx → POST /data/upload → get_project_data_dir(project_name)/
                                                              │
用户提交任务 → ProblemInput.tsx → POST /tasks/submit          │
                                        │                      │
                                        ▼                      ▼
                              PreflightDecisionService.decide()
                                        │
                              DataSchemaExtractor.extract()
                                        │
                              LLM 综合判断 → PreflightReport
                                        │
                    ┌───────────────────┼───────────────────┐
                    ▼                   ▼                   ▼
              sufficient           missing              mismatch
                    │                   │                   │
                    ▼                   ▼                   ▼
              LangGraph           self_collect          422
              Orchestrator         (搜集数据)
```

### 7.2 自主数据收集

当 Preflight 检测到数据缺失时，系统自动：
1. 使用 `research_agent` 搜索相关数据集 URL
2. 调用 `self_collector.collect_urls()` 并发下载（最多 4 并发，50MB 大小限制）
3. 将下载的数据文件加入任务文件列表
4. 重新执行 Preflight 决策

### 7.3 子问题递进求解

**逐个递进模式**（sequential）：

1. `analyzer_agent` 识别子问题列表
2. `modeler_agent` 逐个子问题建模，前序模型结果通过 `previous_models` 递进传递
3. `solver_agent` 逐个子问题求解，前序求解结果通过 `previous_solutions` 递进传递
4. 每个子问题内部有迭代修复循环（最多 `max_solver_iterations` 次）

## 八、前端实时协作

### 8.1 SSE 实时推送

**ChatRoom v2** 支持消息订阅：

```python
class ChatRoom:
    def subscribe(self) -> asyncio.Queue:  # SSE 订阅
    def unsubscribe(self, q: asyncio.Queue):  # 取消订阅
    async def message_stream(self) -> AsyncGenerator[str, None]:  # SSE 流
    def user_post(self, content: str) -> Message:  # 用户发言
```

前端 `AgentChat.tsx` 通过 `POST /{task_id}/messages` 发送用户消息，通过 SSE 实时接收 Agent 发言。

## 十一、系统能力边界与诚实声明

### 11.1 系统能做什么

| 能力 | 状态 | 说明 |
|------|------|------|
| 结构完整的论文生成 | ✅ | 8 套模板，从摘要到参考文献 |
| 真实代码执行 | ✅ | Python 沙箱执行，结果真实 |
| 真实文献检索 | ✅ | arXiv + Semantic Scholar，真实论文 |
| 真实实验执行 | ✅ | GPU 训练，支持小模型（ResNet-18、BERT-base） |
| LaTeX 编译输出 | ✅ | `.cls/.sty` 文件已补齐，可生成 PDF |
| 断点续传 | ✅ | 任务中断后可从 checkpoint 恢复 |
| 显存监控/OOM 防护 | ✅ | VRAMMonitor 实时监控，超阈值自动保护 |
| 检查点保存/恢复 | ✅ | CheckpointManager 自动管理 |
| Human-in-loop | ✅ | 暂停/恢复/编辑，用户可介入 |
| 自主数据收集 | ✅ | Preflight 检测到缺失时自动搜索下载 |
| 数值范围校验 | ✅ | 准确率>1.0 为 error，引用一致性检查 |
| 实验可复现性检查 | ✅ | Peer Review 检查 train 脚本/随机种子/超参数 |

### 11.2 系统不能做什么（诚实声明）

| 限制 | 说明 |
|------|------|
| **无法保证真正的新颖性** | 系统无法与全部已有文献做对比，"novelty" 评分是 LLM 自我评估，非真实审稿人判断 |
| **无法保证实验结果可复现** | 虽然代码真实执行，但 LLM 生成的代码可能有 bug，复杂实验（分布式训练、多 GPU）无法执行 |
| **无法替代研究者的创造性** | 研究想法、方法创新、问题定义仍需人类输入 |
| **大模型训练受限** | 单卡 8GB 显存无法训练 7B+ 参数模型，只能做小模型实验 |
| **引用准确性有限** | 虽然验证 arXiv ID 存在，但无法验证引用内容是否准确对应原文 |
| **LaTeX 编译可能失败** | 生成的 LaTeX 代码可能有语法错误，系统会尝试编译但无法保证 100% 成功 |
| **长程任务稳定性** | 虽然支持断点续传，但长时间运行（>12h）仍可能因系统资源问题中断 |

### 11.3 与 AI Scientist 的差距

| 能力 | AI Scientist (Nature) | 我们的系统 |
|------|----------------------|-----------|
| 真实实验执行 | ✅ 在真实数据集上训练模型 | ✅ 支持，但受限于单卡 8GB 显存 |
| 新颖性验证 | ✅ 与已有论文对比 | ⚠️ LLM 模拟评估，非真实对比 |
| 真实审稿 | ✅ 人类审稿人 | ⚠️ LLM 模拟审稿 |
| 可复现性 | ✅ 代码+数据公开 | ⚠️ 代码公开，但环境配置可能不同 |
| 端到端无人值守 | ✅ 完全自主 | ⚠️ 需要人类设定研究方向和检查关键节点 |

### 11.4 使用建议

1. **作为论文写作辅助工具**：系统可快速生成论文框架、执行标准实验、整理文献，研究者在此基础上修改完善
2. **聚焦轻量级方法**：单卡 8GB 适合模型压缩、高效注意力、小样本学习等方向
3. **人工审核关键节点**：实验设计、核心创新点、主要结论需要研究者确认
4. **多次迭代优化**：利用 peer review + revise 循环逐步提升论文质量
5. **利用自主研究能力**：NAS、AutoML、代码演化等功能可自动发现改进空间，但需人工验证最终方案

## 十二、自主研究能力（v6.0）

### 12.1 神经架构搜索（NAS）

**文件**：`backend/app/core/nas.py`

**核心设计**：
- 基于进化算法的轻量级 NAS，适合单卡 8GB 显存
- Cell-based 搜索空间：每个 cell 包含 4 个节点，每个节点有 2 个前驱边
- 操作池：conv3x3/5x5/7x7、depthwise_conv、separable_conv、dilated_conv、maxpool、avgpool、skip_connection

**算法流程**：
```
初始化种群（随机生成 N 个网络结构）
    │
    ▼
评估适应度（训练每个网络，获取验证准确率）
    │
    ▼
锦标赛选择 + 精英保留
    │
    ▼
变异（操作类型/前驱连接）+ 交叉（单点交换）
    │
    ▼
重复直到收敛或达到最大代数
```

**与 Experiment 节点集成**：CCF-A 模板且问题涉及图像/深度学习时，experiment 节点自动触发 NAS 搜索，将最优架构的 PyTorch 代码注入实验代码。

**参数限制**：
- 默认种群大小：6-8
- 默认最大代数：3-5
- 参数量上限：500万（8GB 显存友好）
- 支持早停：3 代无改进则停止

### 12.2 自动损失函数设计

**文件**：`backend/app/core/loss_design.py`

**核心设计**：
- 基于进化算法的损失函数结构搜索
- 原语库：cross_entropy、focal_loss、MSE、MAE、Huber、log_cosh、加权组合、正则化项
- 表达式树表示：支持嵌套组合（如 weighted_sum(focal_loss, l2_reg)）

**搜索空间**：
| 任务类型 | 可用原语 |
|---------|---------|
| classification | cross_entropy, focal_loss, label_smoothing_ce, weighted_sum, clamp, pow |
| regression | mse, mae, huber, log_cosh, weighted_sum, clamp, pow |
| segmentation | cross_entropy, mse, weighted_sum, clamp, l1_reg, l2_reg |

**与 Experiment 节点集成**：experiment 节点自动检测任务类型，运行损失函数进化搜索，将最优损失函数的 PyTorch 代码注入实验。

### 12.3 跨论文研究空白识别

**文件**：`backend/app/agents/research_agent.py`（`_identify_cross_paper_gaps`）

**核心设计**：
- 在 deep_search / research_paper 模式下自动触发
- 需要至少 3 篇论文才能进行跨论文分析

**算法流程**：
```
提取每篇论文的 limitations / future work
    │
    ▼
LLM 跨论文主题聚类（识别共同局限）
    │
    ▼
统计 frequency（多少篇论文提到该问题）
    │
    ▼
过滤 frequency < 2 的 gap（至少两篇论文提到）
    │
    ▼
生成研究空白报告（severity + novelty_potential）
```

**输出格式**：
```json
{
  "gaps": [
    {
      "description": "现有方法在处理高维稀疏数据时效率低下",
      "frequency": 4,
      "supporting_papers": ["2401.12345", "2402.67890"],
      "severity": "high",
      "novelty_potential": "high"
    }
  ],
  "common_limitations": ["计算复杂度高", "泛化能力有限"],
  "future_directions": ["轻量化设计", "多模态融合"],
  "novelty_opportunities": ["自适应采样策略", "动态网络结构"]
}
```

**与算法工程师集成**：跨论文 gap 分析结果自动注入 `algorithm_engineer_agent` 的上下文，指导其设计针对 gap 的方法。

### 12.4 代码自动演化

**文件**：`backend/app/agents/solver_agent.py`（`CodeEvolutionEngine`）

**核心设计**：
- 求解成功后自动进入演化循环（非一次性生成）
- 基于代码变异的迭代改进：超参数、优化器、激活函数、网络结构、数据增强

**变异策略**：
| 策略 | 变异内容 | 适用场景 |
|------|---------|---------|
| hyperparameters | 学习率、batch_size、epochs、dropout、weight_decay | 所有任务 |
| optimizer | SGD/Adam/AdamW/RMSprop | 深度学习 |
| activation | ReLU/GELU/SiLU/LeakyReLU | 深度学习 |
| architecture | 通道数、层数 | 深度学习 |
| data_augmentation | RandomCrop、RandomHorizontalFlip | 图像任务 |

**演化流程**：
```
初始代码执行成功
    │
    ▼
生成 4 个变体（随机变异）
    │
    ▼
快速评估每个变体（1 次尝试，快速迭代）
    │
    ▼
选择最优变体
    │
    ▼
重复 3 代或达到最大评估次数
```

**与 Solver 节点集成**：`_node_iterative_solver` 在求解成功后自动调用 `evolve_solution`，如果演化后代码性能提升，用最优代码替换原始代码。

### 12.5 AutoML 超参数搜索

**文件**：`backend/app/services/automl.py`

**核心设计**：
- 零外部依赖的纯 Python 实现
- 支持三种搜索策略：TPE（贝叶斯优化）、随机搜索、网格搜索

**TPE 算法**：
```
随机探索阶段（n_startup_trials）
    │
    ▼
按观测值分成好/坏两组（gamma 分位数）
    │
    ▼
为每个超参数拟合核密度估计（KDE）
    │
    ▼
选择使 p_good(x)/p_bad(x) 最大的 x
    │
    ▼
重复直到达到最大迭代次数
```

**搜索空间定义**：
```python
space = SearchSpace()
    .add_log_float("lr", 1e-5, 1e-1)
    .add_int("batch_size", 16, 128)
    .add_categorical("optimizer", ["sgd", "adam", "adamw"])
    .add_float("dropout", 0.0, 0.5)
```

**与 Experiment 节点集成**：experiment 节点从 `algorithm_engineer_agent` 输出的超参数定义自动构建搜索空间，运行 TPE 搜索，将最优超参数注入实验配置。

**自动搜索空间构建**：从 `hyperparameters` 字段自动推断：
- 学习率 → log_float(1e-5, 1e-1)
- batch_size → int(8, 256)
- epochs → int(5, 200)
- dropout → float(0.0, 0.7)
- optimizer → categorical(["sgd", "adam", "adamw", "rmsprop"])
