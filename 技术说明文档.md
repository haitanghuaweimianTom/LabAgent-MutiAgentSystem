## 一、系统定位与技术栈全景

### 1.1 系统定位

本系统是一个**面向 CCF-A 顶会 / 数学建模竞赛的多智能体协作论文全自动生成平台**。系统以自然语言问题描述 + 可选数据文件为输入，由 10+ 个专业化 Agent 组成的团队协作完成从问题分析到 Camera-Ready 论文打包的全流程，最终输出可直接投稿的 LaTeX 论文。

## 二、核心架构设计

### 2.1 LangGraph StateGraph 编排器

**设计动机**：原有 Orchestrator 是硬编码的 if-else 控制流（1500+ 行），新增节点需要修改多处条件分支，维护成本高。LangGraph 的声明式节点-边模型让工作流变更只需修改图结构，不触碰节点内部逻辑。

**状态定义**（`TaskState` TypedDict）：

```python
class TaskState(TypedDict, total=False):
    messages: List[Dict[str, Any]]      # 对话历史
    files: List[str]                     # 数据文件路径
    preflight: Optional[Dict[str, Any]] # Preflight 决策报告
    current_step: str                    # 当前执行步骤
    paper_template: str                  # 论文模板 ID
    workflow_type: str                   # standard/quick/deep_research/code_focused/research_paper
    mode: str                            # batch/sequential
    phase: str                           # 当前阶段
    retry_count: int                     # 重试计数
    escalation_count: int                # 升级计数
    solver_attempts: List[Dict]          # 求解尝试历史
    artifact_paths: List[str]            # 产物路径
    cannot_solve_report: Optional[Dict]  # 无法求解报告
    task_id: str
    problem_text: str
    project_name: Optional[str]
    knowledge_base_id: Optional[str]
    results: Dict[str, Any]              # 所有 Agent 输出汇总
    sub_problems: List[Dict[str, Any]]   # 子问题列表
    should_pause: bool                   # 是否暂停等待用户
    revision_count: int                  # 论文修订轮次
```

**节点图**：

```
preflight_decision ──→ analyzer ──→ data ──→ research ──→ discuss_approach
       │                                                        │
       ├──→ self_collect ──→ preflight_decision (loop)          ▼
       │                                                  modeler ──→ iterative_solver
       └──→ cannot_solve ←──────────────────────────────────────┘    │
              ↑                                                       ├──→ writer ──→ peer_review
              │                                                       │         │        │
              └─── cannot_solve ◄────────────────────────────────────┘    ├──→ fact_check ──→ END
                                                  ├─→ wait_user ──→ END   │
                                                  └─→ revise (loop) ◄────┘
```

**条件边**：

| 源节点 | 路由函数 | 分支 |
|--------|---------|------|
| `preflight_decision` | `_route_preflight` | standard / quick / deep_research / code_focused / research_paper / self_collect / abort |
| `peer_review` | `_route_peer_review` | accept / revise / wait_user / abort |
| `iterative_solver` | `_route_solver` | success / retry / escalate / abort |
| `research` | `_route_after_research` | discuss / proceed |

**兼容层**：`orchestrator.py` 的 `execute_workflow` 检测 `use_langgraph_orchestrator=True` 后委托给 `LangGraphOrchestrator.run()`，旧路径保留不变，通过 `config.py` 的 `use_langgraph_orchestrator` 开关切换。

### 2.2 Preflight 决策引擎

**设计动机**：原有系统在 `submit_task` 时不做任何数据/流程判断，直接跑 phase1。导致两个问题：
1. 无数据时 LLM 凭空编造数值（幻觉）
2. 问题类型不匹配时走错工作流

**决策流程**：

```
用户提交任务
    │
    ▼
DataSchemaExtractor.extract()
    │  提取：行数/列数/列类型/数值范围/缺失率
    ▼
LLM 综合判断（单次调用，ReAct-style）
    │  输出：problem_type / has_data_confidence / data_subjects
    │        recommended_template / recommended_workflow / recommended_mode
    │        data_adequacy / llm_should_collect / collection_plan
    ▼
┌─ data_adequacy == "sufficient" ──→ 正常启动
├─ data_adequacy == "missing" + llm_should_collect ──→ self_collect
├─ data_adequacy == "missing" + 无法搜集 ──→ 422 要求上传
└─ data_mismatch_warning ──→ 422 + 报告
```

**关键数据结构**（`PreflightReport`）：

```python
@dataclass
class PreflightReport:
    problem_type: str                    # 优化/预测/评价/分类/仿真/网络/物理/测量/综合
    has_data_confidence: float           # 0-1，数据与题目的匹配度
    data_subjects: List[str]             # 数据主题（金融时序/医学影像/工程测量/...）
    recommended_template: str            # 8 模板之一
    recommended_workflow: str            # standard/quick/deep_research/code_focused/research_paper
    recommended_mode: str                # batch/sequential
    data_adequacy: DataAdequacy          # sufficient/insufficient/missing
    llm_should_collect: bool             # 是否建议自主搜集
    collection_plan: str                 # 搜集计划（搜什么、去哪搜）
    data_mismatch_warning: Optional[str] # 数据不匹配警告
    data_schemas: List[Dict]             # 每份数据的 schema 摘要
```

### 2.3 ReAct 工具循环

**设计动机**：原有 Agent 的 `call_llm` 是单次调用，LLM 无法在推理过程中使用工具（搜索文献、读取数据、执行代码）。引入 ReAct 循环后，Agent 可以在 Thought-Action-Observation 周期中自主决策工具使用。

**实现架构**：

```python
async def call_llm(self, messages, tools=None, max_react_iterations=5, ...):
    if not tools:
        return await self._call_llm_once(messages, temperature)

    # ReAct 循环
    for iteration in range(max_react_iterations):
        response = await self._call_llm_once(messages, temperature, tools=tools_payload)
        tool_calls = self._parse_tool_calls(response, api_format)
        if not tool_calls:
            return response  # LLM 给出最终答案

        # 执行工具，拼回 messages
        for tc in tool_calls:
            observation = await self._execute_tool_call(tc)
            messages.append(self._build_tool_result_message(tc, observation, api_format))
```

**工具注册表**（5 种内置工具）：

| 工具名 | 功能 | 实现 |
|--------|------|------|
| `search_web` | 文献搜索 | 接入 `research_agent.execute()` |
| `read_csv_columns` | 读取数据列信息 | 接入 `DataSchemaExtractor.extract()` |
| `run_python` | 沙箱执行 Python | `subprocess.run` + 30s timeout |
| `write_python_snippet` | 写文件 | `Path.write_text` |
| `read_paper` | 读取论文元数据 | 接入 Semantic Scholar + MetadataCache |

**Provider 适配**：

| API 格式 | tools 格式 | tool_calls 解析 |
|----------|-----------|----------------|
| `anthropic` | `{"name", "description", "input_schema"}` | `content[].type == "tool_use"` |
| `openai_chat` | `{"type": "function", "function": {...}}` | `message.tool_calls[]` |
| `ollama_chat` | 同 OpenAI | 同 OpenAI |
| `gemini_native` | 暂不支持 | 忽略 tools，走普通调用 |

## 三、多智能体协作机制

### 3.1 Agent 团队

| Agent | 角色 | 核心能力 | 默认模型 |
|-------|------|---------|---------|
| `analyzer_agent` | 问题分析师 | 子问题分解、问题类型识别、难度评估 | sonnet |
| `data_agent` | 数据分析师 | 数据文件解析、schema 提取、洞察抽取 | sonnet |
| `research_agent` | 研究员 | arXiv 检索 + Semantic Scholar 增强 + Reranker | haiku |
| `modeler_agent` | 建模师 | 数学建模、模型选择、算法推荐 | sonnet |
| `solver_agent` | 求解器 | 代码生成 + 沙箱执行 + 自动修复 + 跨方法验证 | sonnet |
| `writer_agent` | 写作专家 | 按章节独立 LLM 调用、LaTeX 生成、自评 critique | opus |
| `peer_review_agent` | 审稿人 | 4 维评分（novelty/soundness/clarity/significance） | sonnet |
| `experimentation_agent` | 实验设计 | baselines/datasets/metrics/ablation plan | haiku |
| `coordinator` | 协调者 | 工作流编排、暂停/恢复、跨 Agent 黑板 | - |

**AgentModelRouter**：按 `(agent_name, template_id)` 路由模型。CCF-A 4 套模板强制 `writer = opus`。

### 3.2 多 Agent 讨论协议

**设计动机**：原有系统是线性数据流（A → B → C），Agent 之间没有真正的"讨论"。引入讨论节点后，多个 Agent 可以在关键决策点互相审阅、提出建议。

**`discuss_approach` 节点**：

```
分析师: "我认为应该用线性规划，因为..."
建模师: "同意，但需要考虑约束的非线性..."
研究员: "arXiv 上有一篇类似问题用混合整数规划..."
协调者: "综合各方意见，采用混合整数规划方案。"
```

**用户参与机制**：

- 前端 AgentChat 组件支持用户输入消息（`POST /{task_id}/messages`）
- 用户消息实时出现在 Agent 讨论流中（SSE 推送）
- `_route_peer_review`：用户已发言 → `wait_user`（等待用户指导）；用户未发言 → 自动修订（最多 3 轮）

### 3.3 自主迭代求解

**设计动机**：借鉴 Claude Code Plan mode 的"自主迭代改进直到求出答案"理念。solver 失败后不是立即停止，而是在 Harness 评判下自动修复，直到成功或集体判定无法求解。

**迭代流程**：

```
solver_agent.execute()
    │
    ▼
Harness 评判（ResultValidator + CrossValidator + CodeManifest）
    │
    ├── passed ──→ 进入下一子问题
    │
    └── failed ──→ _classify_execution_error
                    │  注入错误分类 + 修复建议到下一轮 prompt
                    ▼
              solver_agent.execute()（第 2 次）
                    │
                    ├── passed ──→ 进入下一子问题
                    │
                    └── failed ──→ ...（最多 max_solver_iterations 次）
                                    │
                                    ▼
                              _multi_agent_vote
                                    │
                    ┌───────────────┼───────────────┐
                    ▼               ▼               ▼
                  retry        collect_data       abort
                  (升级重试)    (搜集数据)       (无法求解)
```

**多 Agent 投票**（`_multi_agent_vote`）：

向 `analyzer_agent`、`modeler_agent`、`peer_review_agent` 发送诊断 prompt，每个 Agent 独立判断失败原因（代码 bug / 数据不足 / 无法继续），多数投票决定后续行为。

## 四、Harness 校验体系（5 层验证）

**设计动机**：论文中的数值必须有真实来源，不能靠 LLM 幻觉。5 层校验从不同维度确保结果的正确性。

| 层级 | 服务 | 文件 | 校验内容 |
|------|------|------|---------|
| L1 | ResultValidator | `services/result_validator.py` | NaN/Inf/负数/范围/一致性 |
| L2 | CrossValidator | 同上 | 5% 阈值跨方法比对 |
| L3 | CodeManifest | `services/code_manifest.py` | 300 行/多角色/多子问题拆分校验 |
| L4 | ContractValidator | `services/contract_validator.py` | Pydantic schema 校验 Agent 输出 |
| L5 | FactChecker | `services/fact_checker.py` | main.tex 数字与 solves.json 对账 |

**ResultValidator 问题类型**：

| level | type | 描述 |
|-------|------|------|
| error | `empty` | 结果为空 |
| error | `nan` | 含 NaN |
| error | `inf` | 含 Inf |
| error | `negative` | 非负字段为负 |
| error | `range` | R² > 1.0 |
| warning | `magnitude` | 数值过大/过小 |
| warning | `consistency` | AHP 的 CR ≥ 0.1 |

**CrossValidator**：跨方法比对，差异 > 5% 自动报警。支持注册 `ALTERNATIVE_METHODS`（`analytical_estimate` / `llm_second_opinion`）。

**FactChecker**：从 `main.tex` 提取所有数字，与 `solves.json` 比对，超阈值报警。确保论文中的每个数值都有真实来源。

## 五、多智能体记忆与幻觉抑制

### 5.1 设计动机与基础架构

#### 5.1.1 三级记忆架构

灵感来自 **G-Memory: Tracing Hierarchical Memory for Multi-Agent Systems** (NeurIPS 2026)。系统采用三层记忆：

```
MemoryManager
├── Working Memory（任务级黑板）
│   ├── problem: {text, type, template, difficulty}
│   ├── sub_problems: List[Dict]
│   ├── data_insights: List[str]
│   ├── literature: List[Dict]
│   ├── methods: List[Dict]
│   ├── agent_results: Dict[str, Any]
│   └── decisions: List[Dict]
│
├── Episodic Memory（事件流）
│   └── events: List[{agent, action, detail, timestamp}]
│
└── Lessons Memory（跨任务经验）
    └── lessons: List[{problem_type, lesson, use_count, score}]
```

**Working Memory**：任务级黑板，各 Agent 读写中间结果。`get_context_for_agent(agent_name)` 按角色过滤上下文，避免信息过载。

**Episodic Memory**：记录每个 Agent 的执行事件，支持 `compress()` 压缩（保留最近 50 条）。

**Lessons Memory**：跨任务经验提取。任务完成后自动 `extract_lessons_from_result`，下次任务用 `retrieve_relevant` 检索相关经验。

#### 5.1.2 RAG 知识库

- TF-IDF 语义检索（零依赖回退）
- Sentence-Transformer 嵌入（可选）
- RESTful API：文档 CRUD / 语义查询 / Agent 自动注入
- Agent 构建 Prompt 时自动查询知识库并注入相关上下文

#### 5.1.3 三大核心问题

随着系统运行时间增长，三级记忆（Working / Episodic / Lessons）面临三个核心问题：

1. **记忆爆炸**：Lessons 条目无上限增长，Working Memory 内容膨胀导致 prompt 超限，checkpoint 文件体积膨胀拖慢恢复速度。
2. **上下文溢出**：单次 LLM 调用携带的黑板状态 + 知识库 + 历史经验 + ReAct 循环 messages 很容易超过 128k 上下文窗口，导致模型截断或性能骤降。
3. **幻觉累积**：LLM 在文献引用、数值结果、跨 Agent 传递中可能产生虚构信息，且随着迭代轮次增加，错误会被后续 Agent 当作事实继续传播。

本章节阐述系统如何通过**独立 Agent 记忆**、**共享黑板版本控制**、**Token 预算管理**、**语义压缩**和**置信度评分体系**解决上述问题。

### 5.2 独立 Agent 记忆（AgentProfileMemory）

#### 5.2.1 数据模型

每个 Agent 拥有独立的 `AgentProfileMemory`，包含以下维度：

```python
class AgentProfileMemory:
    work_style: str                          # Agent 工作风格描述
    success_cases: List[AgentCase]          # 成功案例
    failure_cases: List[AgentCase]           # 失败案例
    preferences: AgentPreferences            # 偏好配置
    skill_inventory: List[AgentSkill]      # 技能清单
    collaboration_notes: List[str]           # 协作备注

class AgentCase:
    case_id: str
    task_id: str
    problem_type: str
    method: str
    outcome: str
    impact_score: float       # 重要性 0-1
    recency_score: float      # 时间衰减 0-1
    embedding: List[float]    # 语义向量（SentenceTransformer）

class AgentPreferences:
    model_type: str
    temperature: float
    max_tokens: int
    tool_preferences: List[str]
    output_format: str

class AgentSkill:
    skill_name: str
    proficiency: float       # 熟练度 0-1
    last_used: str           # ISO 时间
    usage_count: int
```

#### 5.2.2 生命周期

**收集**：任务完成后由 `LessonExtractor` 提取 Agent 专属经验，结合 `ResultValidator` 和 `PeerReview` 结果自动判定成功/失败。用户反馈通过 `POST /agents/{name}/cases/feedback` 直接关联到特定 Agent。

**更新**：
- 每次任务后自动 `append` 新案例
- 每 N 次任务（默认 5 次）触发 LLM 对案例进行语义摘要和合并
- 重要性评分公式：`impact_score = success_score × problem_complexity × user_rating`

**检索**：
- 按 `problem_type` 向量语义匹配（SentenceTransformer 嵌入）
- 按 `method` 名称精确匹配
- 时间衰减：`recent_weight = exp(-days_since / 30)`
- 综合排序：`score = similarity × importance × recency`

**遗忘**：
- 案例数量上限：100 成功 + 100 失败
- `importance < 0.2` 且 `TTL > 90` 天自动归档到 `{agent_name}_archive.json`
- 归档案例不再参与检索但可手动查询

#### 5.2.3 自动注入

`BaseAgent.call_llm` 自动注入 `agent_profile` 的 `preferences`（温度、模型偏好）和 `top_k` 个最相关的案例（默认 3 个）。注入格式：

```
【Agent 个人经验】
成功案例：...
失败教训：...
偏好配置：...
```

#### 5.2.4 存储

- 每个 Agent 一个 JSON 文件：`data/memory/agents/{agent_name}.json`
- 归档案例单独存储：`data/memory/agents/{agent_name}_archive.json`
- 进程内缓存：`MemoryManager._agent_profiles` 字典，LRU 淘汰最久未访问的 Agent 配置

### 5.3 共享记忆黑板（Shared Blackboard）

#### 5.3.1 数据模型扩展

`WorkingMemory` 扩展以下字段：

```python
class WorkingMemory:
    _write_permissions: Dict[str, List[str]]     # 每个 Agent 可写的分区
    _version_history: Dict[str, List[VersionSnapshot]]  # 每个 key 的修改历史
    _max_versions_per_key: int = 10

class VersionSnapshot:
    version: int
    value: Any
    agent: str
    timestamp: str
    operation: str   # set / append / update
```

#### 5.3.2 访问控制矩阵

| Agent | 可写分区 | 可读分区 |
|-------|---------|---------|
| `research_agent` | literature, methods | 所有 result 摘要 |
| `analyzer_agent` | problem, constraints, sub_problems | 所有 result 摘要 |
| `modeler_agent` | methods（追加） | 所有 result 摘要 |
| `solver_agent` | results.{agent_name} | 所有 result 摘要 |
| `writer_agent` | decisions（追加） | 所有 result 摘要 |
| 所有 Agent | notes（支持 @mention 定向） | notes（定向 + 广播） |
| `coordinator` | 所有分区 | 所有分区 |
| `user` | notes, decisions | notes, decisions |

**读取过滤**：`get_context_for_agent` 按角色过滤，如 `modeler_agent` 看不到 `writer_agent` 的完整 `latex_code`，只能看到摘要。

#### 5.3.3 版本控制与一致性

- 每个 key 的修改历史保留最近 10 个版本（version counter 递增）
- 写操作：copy-on-write 语义，读取时获取 snapshot
- 冲突解决：last-write-wins，记录冲突日志到 `blackboard_versions/{task_id}.json`
- 支持 rollback：`POST /task/{task_id}/blackboard/rollback` 回退到指定版本
- 一致性保证：单任务内串行写入（LangGraph 节点顺序执行），无并发写冲突

#### 5.3.4 大对象分层存储

黑板实时状态：`MemoryManager._working` 字典（进程内，LRU 淘汰）。
版本历史：`data/memory/blackboard_versions/{task_id}.json`（按需加载）。
大对象引用：`data/tasks/{task_id}_blobs.json`（`latex_code`、长代码等超过 10KB 的对象）。
任务完成后：WorkingMemory 摘要版本持久化到 `task_{task_id}.json`，完整版本可配置保留或删除。

### 5.4 长上下文管理

#### 5.4.1 三层上下文层级

| 层级 | 组件 | 内容 | Token 预算占比 |
|------|------|------|---------------|
| L1 | WorkingMemory / 黑板 | 当前任务实时状态 | 40% |
| L2 | EpisodicMemory | 时间戳化事件记录 | 20% |
| L3 | AgentProfileMemory + LessonsMemory | 跨任务持久化经验 | 15% |
| - | 用户查询 | 原始问题 + 指令 | 25% |

#### 5.4.2 Token 预算管理（TokenBudgetManager）

单例模式，根据模型类型动态设置总预算：

| 模型 | 总预算 |
|------|--------|
| claude-sonnet | 200k |
| gpt-4 | 128k |
| 其他 | 128k（默认） |

预算分配：
- `system_prompt`：20%
- `user_query`：30%
- `knowledge_context`：15%
- `memory_context`：15%
- `agent_profile`：10%
- `react_history`：10%

**溢出处理**：每部分超出预算时自动裁剪：
- `knowledge`：减少 `top_k`（最低 1）
- `memory`：减少摘要长度（最低 100 字符）
- `agent_profile`：只保留最相关 1 个案例
- `react_history`：压缩最早轮次

**接口**：`reserve_budget(category, tokens)` / `release_budget(category)`。`call_llm` 入口处检查总预算，超出时抛出 `ContextOverflowException`，由 Orchestrator 决策：拆分任务、升级模型或请求用户确认。

#### 5.4.3 语义压缩（ContextCompressor）

LLM 驱动的语义摘要服务，支持以下场景的自动压缩：

**ReAct 循环**：messages 超过 80% 上下文窗口时，对最早的 `tool_call + observation` 对进行摘要压缩：保留 `tool_name` 和 `observation` 前 200 字符，其余用 LLM 生成一句话摘要。系统消息和用户原始问题永远保留。

**Sequential 模式**：每 3 个子问题后调用 LLM 对 `previous_model_summary` / `previous_solution_summary` 进行合并压缩，保留子问题名称、核心方法/算法、关键数值结果（最多 3 个）。

**EpisodicMemory**：每 10 个 entries 自动触发 LLM 摘要，摘要后原始 entries 只保留最近 5 个。

**自动摘要触发条件**：
- EpisodicMemory.entries 达到 10 条
- sequential 模式每 3 个子问题
- ReAct 循环每 3 轮
- chat_room 消息每 50 条

**摘要策略**：保留关键实体（Agent 名称、决策、数值结果、错误类型），删除冗余描述和中间推导过程。摘要格式：
```
[Agent] 在 [task] 中采用 [method]，结果为 [outcome]，关键教训：[lesson]
```

### 5.5 记忆爆炸控制

#### 5.5.1 存储上限

| 组件 | 上限 | 超出策略 |
|------|------|---------|
| AgentProfileMemory 成功案例 | 100 条 | 按 impact_score + recency 淘汰 |
| AgentProfileMemory 失败案例 | 100 条 | 同上 |
| EpisodicMemory max_entries | 30（从 50 下调） | 滚动窗口 |
| EpisodicMemory content 截断 | 300 字符（从 500 下调） | 硬截断 |
| MemoryManager._working 字典 | 100 个任务 | LRU 淘汰最旧任务 |
| KnowledgeManager._kb_instances | 5 个实例 | LRU 淘汰 |
| VectorStore 单个知识库 | 10,000 文档 | 拒绝新增 |
| MetadataCache | 1000 条 | LRU 淘汰 |
| MetadataCache 单条大小 | 50KB | 只保留核心字段 |
| Checkpoint 单个文件 | 100KB | 自动分层存储 |
| ChatRoom 消息 | 500 条 | 归档到 `chat_archive/{task_id}.jsonl` |

#### 5.5.2 归档策略

- **AgentCase**：超过 100 条时，`impact_score < 0.2` 且 `TTL > 90` 天归档到 `{agent_name}_archive.json`
- **LessonsMemory**：TTL 过期（默认 180 天）且 `use_count < 3` 的条目归档到 `lessons_archive.db`
- **ChatRoom**：消息超过 500 条时保留最近 100 条，其余归档到 `data/memory/chat_archive/{task_id}.jsonl`
- **Checkpoint**：同任务超过 20 个时，合并最旧的 5 个

#### 5.5.3 去重策略

- **LessonsMemory**：新增内容时计算 embedding 向量相似度，与已有条目比对，`similarity > 0.85` 视为重复，合并 `use_count` 并更新内容（取更详细的版本）
- **AgentCase**：按 `(problem_type, method, outcome)` 三元组去重，相同组合只保留 `impact_score` 最高和最近的两条
- **知识库**：文档添加时与已有文档比对相似度，`> 0.9` 时合并分块而非新增

#### 5.5.4 TTL 策略

| 数据类型 | 默认 TTL | 延长条件 |
|---------|---------|---------|
| LessonsMemory | 180 天 | `use_count > 10` 延长到 365 天 |
| AgentCase | 90 天 | `impact_score > 0.8` 延长到 180 天 |
| MetadataCache | 7 天 | - |
| EpisodicMemory 摘要后原始 entries | 30 天 | - |
| ChatRoom 归档消息 | 90 天 | - |

**过期处理**：TTL 过期不立即删除，而是降低检索权重（`weight *= 0.5`），再过一个 TTL 周期后归档。

#### 5.5.5 重要性评分体系

- **AgentCase**：`impact_score = success_score × problem_complexity × user_rating`（0-1）
- **LessonsMemory**：`importance_score = use_count × recency_weight × user_rating`
- **EpisodicMemory**：按 `event_type` 赋予基础权重（`error=1.0`, `decision=0.8`, `complete=0.5`, `note=0.3`）

重要性评分用于排序和遗忘决策。

### 5.6 幻觉抑制体系

#### 5.6.1 文献引用验证（CitationVerifier）

对 `research_agent` 返回的每篇论文验证真实性：

- **DOI 验证**：通过 Semantic Scholar API `paper/DOI:{doi}` 查询
- **arXiv ID 验证**：通过 `arxiv.org/abs/{id}` HTTP HEAD 请求
- **验证状态**：`verified=1.0` / `unverified=0.3` / `phantom=0.0`

`writer_agent` 生成 LaTeX 时，`FactChecker` 增加 `citation_check` 方法：
- 提取所有 `\cite{}` 和 `\citep{}`
- 与 `CitationRegistry`（任务级注册表）比对
- 未找到的标记为 "phantom citation" 并生成 warning

**Mock 结果标记**：
- 所有 mock 结果统一增加 `"_source": "mock"` 标记
- `ResultValidator` 跳过 mock 数值验证，UI 显示 "演示模式"
- `solver_agent` mock 占位符（如 `"最优解": "待运行代码"`）增加 `"_placeholder": true`
- `writer_agent` 跳过 placeholder 数值，用文字描述替代
- `peer_review_agent` `_empty_review` 的 `recommendation` 改为 `"needs_review"`，`confidence=0` 触发重新调用

#### 5.6.2 置信度评分系统（ConfidenceScoringSystem）

每个信息单元（文献、数值、经验）都有 `ConfidenceScore`：

```python
class ConfidenceScore:
    source_type: str       # llm / knowledge_base / agent_memory / human / verified_api
    raw_score: float       # 0-1
    verification_chain: List[str]   # 来源追溯链
    cross_validation_count: int     # 交叉验证通过次数
    human_confirmed: bool           # 用户是否确认
```

**最终评分公式**：
```
final_score = raw_score × min(1.0, 0.5 + 0.25 × cross_validation_count)
```
若 `human_confirmed=True` 则 `+0.2`，上限 1.0。

**关键阈值**：
- `final_score < 0.5`：标记为 "low_confidence"，在 UI 中显示警告
- `final_score < 0.3`：拒绝使用该信息

#### 5.6.3 多 Agent 交叉验证（MultiAgentVerification）

关键结论需要至少 2 个 Agent 交叉确认：

- **问题类型判断**：`analyzer_agent` 和 `modeler_agent` 对问题类型判断不一致时触发仲裁（由 `coordinator` 调用 LLM 判定）
- **数值结果验证**：`solver_agent` 的数值结果与 `CrossValidator` 的 `analytical_estimate` 比对，差异 > 5% 时标记为 "needs_verification"
- **论文数字对账**：`writer_agent` 的 LaTeX 数字与 `solver_agent` 的 `solves.json` 通过 `FactChecker` 对账

所有比对结果写入 `ConfidenceScore.cross_validation_count`。

#### 5.6.4 检索验证增强

- **LessonsMemory**：从关键词匹配升级为向量语义检索（`sqlite-vec` 或 `faiss`），增加 CrossEncoder 精排，`similarity < 0.6` 的结果不返回
- **知识库注入**：增加免责声明："以下信息来自知识库，请交叉验证"，每个知识库条目标记 `source_trust` 评分
- **Preflight LLM 决策**：增加枚举校验和 fallback 机制（推荐不存在模板时回退到 `math_modeling`），`confidence_score` 基于 prompt 与模板描述相似度计算

### 6.2 DataSchemaExtractor

从 CSV/Excel/JSON 文件中提取结构化 schema：

```python
{
    "file_name": "data.csv",
    "row_count": 1000,
    "column_count": 5,
    "columns": [
        {
            "name": "price",
            "dtype": "float64",
            "nullable": False,

            "unique_count": 856,

        }
    ]

```

### 6.3 子问题递进求解

**逐个递进模式**（sequential）：

1. `analyzer_agent` 识别子问题列表（正则预提取 + LLM 分析 + 关键词兜底）
2. `modeler_agent` 逐个子问题建模，前序模型结果通过 `previous_models` 递进传递
3. `solver_agent` 逐个子问题求解，前序求解结果通过 `previous_solutions` 递进传递
4. 每个子问题内部有迭代修复循环（最多 `max_solver_iterations` 次）

**递进传递示例**：

```
子问题1 建模 → model_1（LP_Model, min c^T x）
子问题1 求解 → solution_1（optimal_value=42.0）
    │
    ▼ 递进传递
子问题2 建模 → model_2（看到 model_1 的结果，选择更合适的方法）
子问题2 求解 → solution_2（看到 solution_1 的数值，调整参数）
```
