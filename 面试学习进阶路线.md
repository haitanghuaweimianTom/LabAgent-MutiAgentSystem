- `asyncio.gather()` vs `asyncio.create_task()` vs `await`

**对照你的项目代码**：

```python
# 你的项目中大量使用 async/await，面试官会追问原理

# 1. 基本用法
async def run_agent(agent, context):
    result = await agent.run(context)  # 挂起，让出事件循环
    return result

# 2. 并发执行多个 Agent
async def run_parallel(agents, context):
    tasks = [agent.run(context) for agent in agents]
    results = await asyncio.gather(*tasks)  # 并发执行
    return results

# 3. 用 Semaphore 控制并发数（你的项目用到了）
sem = asyncio.Semaphore(3)  # 最多 3 个并发

async def limited_run(agent, context):
    async with sem:
        return await agent.run(context)

# 面试官会问：asyncio.Semaphore 的原理是什么？
# 答：底层是一个计数器 + 等待队列。acquire 时计数器-1，
# 如果 <0 则加入等待队列；release 时计数器+1，唤醒队首。
```

**面试追问**：
- Q: "asyncio.gather 和 asyncio.create_task 有什么区别？"
  - A: "`gather` 是一次性提交所有任务并等待全部完成；`create_task` 是创建单个 Task 对象，可以更灵活地控制（cancel、加回调）。项目中用 `gather` 是因为需要所有 Agent 结果一起处理。"

- Q: "如果一个 Agent 的 LLM 调用卡住了（网络超时），会不会阻塞其他 Agent？"
  - A: "不会。asyncio 是单线程协作式调度，每个 `await` 点都是让出控制权的机会。只要 LLM 调用是异步的（用了 httpx 异步客户端），一个 Agent 的 I/O 等待不会阻塞其他 Agent。但如果在 async 函数中调用了同步的 `time.sleep()` 或 `requests.get()`，就会阻塞整个事件循环。"

- Q: "Python 的 asyncio 和 JavaScript 的 Promise 有什么关系？"
  - A: "都是异步编程模型，但实现不同。JS 的 Promise 是基于回调的链式调用（.then），Python 的 asyncio 是基于生成器的协程（async/await）。asyncio 需要显式启动事件循环（`asyncio.run()`），JS 的事件循环是运行时内置的。"

**对照你的项目代码**：

```python
# 你的项目中用到的装饰器模式

# 1. 重试装饰器
import functools

def retry(max_retries=3, delay=1):
    def decorator(func):
        @functools.wraps(func)  # 保留原函数的 __name__ 等属性
        async def wrapper(*args, **kwargs):
            for attempt in range(max_retries):
                try:
                    return await func(*args, **kwargs)
                except Exception as e:
                    if attempt == max_retries - 1:
                        raise
                    await asyncio.sleep(delay * (attempt + 1))
        return wrapper
    return decorator

# 使用
@retry(max_retries=3, delay=2)
async def call_llm(prompt):
    ...

# 面试官会问：为什么需要 functools.wraps？
# 答：没有它的话，装饰后的函数 __name__ 会变成 'wrapper' 而不是原函数名，
# 导致调试困难、日志不正确、某些依赖 __name__ 的框架出错。

# 2. 异步装饰器
def log_execution(func):
    @functools.wraps(func)
    async def wrapper(*args, **kwargs):
        print(f"Starting {func.__name__}")
        result = await func(*args, **kwargs)
        print(f"Finished {func.__name__}")
        return result
    return wrapper
```

**面试追问**：
- Q: "写一个装饰器，统计异步函数的执行时间"
  - A: 见上方 `log_execution` 的变体，加 `time.time()` 计时。
- Q: "装饰器能修改函数的返回值吗？"
  - A: "能。装饰器本质上是高阶函数，包装了原函数，可以在调用前后修改参数和返回值。比如项目的 `_validate_no_fabrication` 就是在 Agent 输出后添加校验标记。"

```python
# 你的熔断器中的异常处理
class CircuitBreakerError(Exception):
    """熔断器触发时抛出"""
    pass

class CircuitBreaker:
    def __init__(self, failure_threshold=10, recovery_timeout=300):
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.state = "CLOSED"  # CLOSED → OPEN → HALF_OPEN
        self.failure_count = 0
        self.last_failure_time = None
    
    async def call(self, func, *args, **kwargs):
        if self.state == "OPEN":
            if time.time() - self.last_failure_time > self.recovery_timeout:
                self.state = "HALF_OPEN"
            else:
                raise CircuitBreakerError("Circuit breaker is OPEN")
        
        try:
            result = await func(*args, **kwargs)
            if self.state == "HALF_OPEN":
                self.state = "CLOSED"
                self.failure_count = 0
            return result
        except Exception as e:
            self.failure_count += 1
            self.last_failure_time = time.time()
            if self.failure_count >= self.failure_threshold:
                self.state = "OPEN"
            raise  # 重新抛出，让调用方处理
```

### Day 3：JSON 序列化与文件 I/O

**你的项目中大量读写 JSON（checkpoint、记忆、模板）**：

```python
import json
import os
import tempfile
from pathlib import Path

# 1. 原子写入 — 你的 task_persistence.py 用到的模式
def atomic_write_json(filepath: Path, data: dict):
    """原子写入 JSON，防止写入过程中断电损坏"""
    # 先写临时文件
    tmp_fd, tmp_path = tempfile.mkstemp(dir=filepath.parent, suffix=".tmp")
    try:
        with os.fdopen(tmp_fd, 'w') as f:
            json.dump(data, f, ensure_ascii=False, indent=2)
            f.flush()
            os.fsync(f.fileno())  # 强制刷盘
        # 原子替换（在同一文件系统内是原子操作）
        os.replace(tmp_path, filepath)
    except Exception:
        os.unlink(tmp_path)  # 清理临时文件
        raise

# 2. 文件锁 — 防止并发写入
import fcntl

def safe_write_json(filepath: Path, data: dict):
    with open(filepath, 'w') as f:
        fcntl.flock(f.fileno(), fcntl.LOCK_EX)  # 排他锁
        try:
            json.dump(data, f, ensure_ascii=False, indent=2)
        finally:
            fcntl.flock(f.fileno(), fcntl.LOCK_UN)  # 释放锁

# 3. 大 JSON 的流式读取
def stream_large_json(filepath: Path):
```

```python
# 快排 — 最常考
def quicksort(arr):
    if len(arr) <= 1:
        return arr
    pivot = arr[len(arr) // 2]
    left = [x for x in arr if x < pivot]
    middle = [x for x in arr if x == pivot]
    right = [x for x in arr if x > pivot]
    return quicksort(left) + middle + quicksort(right)

# 原地快排（面试更常考）
def quicksort_inplace(arr, low, high):
    if low < high:
        pivot_idx = partition(arr, low, high)
        quicksort_inplace(arr, low, pivot_idx - 1)
        quicksort_inplace(arr, pivot_idx + 1, high)

def partition(arr, low, high):
    pivot = arr[high]
    i = low - 1
    for j in range(low, high):
        if arr[j] <= pivot:
            i += 1
            arr[i], arr[j] = arr[j], arr[i]
    arr[i + 1], arr[high] = arr[high], arr[i + 1]
    return i + 1

# 归并排序
def mergesort(arr):
    if len(arr) <= 1:
        return arr
    mid = len(arr) // 2
    left = mergesort(arr[:mid])
    right = mergesort(arr[mid:])
    return merge(left, right)

def merge(left, right):
    result = []
    i = j = 0
    while i < len(left) and j < len(right):
        if left[i] <= right[j]:
            result.append(left[i]); i += 1
        else:
            result.append(right[j]); j += 1
    result.extend(left[i:])
    result.extend(right[j:])
    return result

# 复杂度对比
# | 算法 | 最优 | 平均 | 最差 | 空间 | 稳定 |
# |------|------|------|------|------|------|
# | 快排 | O(nlogn) | O(nlogn) | O(n²) | O(logn) | 否 |
# | 归并 | O(nlogn) | O(nlogn) | O(nlogn) | O(n) | 是 |
# | 堆排 | O(nlogn) | O(nlogn) | O(nlogn) | O(1) | 否 |
```

```python
def knapsack(weights, values, capacity):
    n = len(weights)
    dp = [[0] * (capacity + 1) for _ in range(n + 1)]
    for i in range(1, n + 1):
        for w in range(1, capacity + 1):
            dp[i][w] = dp[i-1][w]  # 不选第 i 件
            if weights[i-1] <= w:
                dp[i][w] = max(dp[i][w], 
                    dp[i-1][w - weights[i-1]] + values[i-1])
    return dp[n][capacity]

# 2. 最长公共子序列
def lcs(text1, text2):
    m, n = len(text1), len(text2)
    dp = [[0] * (n + 1) for _ in range(m + 1)]
    for i in range(1, m + 1):
        for j in range(1, n + 1):
            if text1[i-1] == text2[j-1]:
                dp[i][j] = dp[i-1][j-1] + 1
            else:
                dp[i][j] = max(dp[i-1][j], dp[i][j-1])
    return dp[m][n]

# 3. 编辑距离
def min_distance(word1, word2):
    m, n = len(word1), len(word2)
    dp = [[0] * (n + 1) for _ in range(m + 1)]
    for i in range(m + 1): dp[i][0] = i
    for j in range(n + 1): dp[0][j] = j
    for i in range(1, m + 1):
        for j in range(1, n + 1):
            if word1[i-1] == word2[j-1]:
                dp[i][j] = dp[i-1][j-1]
            else:
                dp[i][j] = 1 + min(dp[i-1][j], dp[i][j-1], dp[i-1][j-1])
    return dp[m][n]
```

```python
# BFS 最短路径
from collections import deque
def bfs_shortest_path(graph, start, end):
    queue = deque([(start, 0)])
    visited = {start}
    while queue:
        node, dist = queue.popleft()
        if node == end:
            return dist
        for neighbor in graph[node]:
            if neighbor not in visited:
                visited.add(neighbor)
                queue.append((neighbor, dist + 1))
    return -1

# 拓扑排序（DAG）
from collections import deque
def topological_sort(numCourses, prerequisites):
    graph = {i: [] for i in range(numCourses)}
    in_degree = [0] * numCourses
    for dest, src in prerequisites:
        graph[src].append(dest)
        in_degree[dest] += 1
    
    queue = deque([i for i in range(numCourses) if in_degree[i] == 0])
    order = []
    while queue:
        node = queue.popleft()
        order.append(node)
        for neighbor in graph[node]:
            in_degree[neighbor] -= 1
            if in_degree[neighbor] == 0:
                queue.append(neighbor)
    return order if len(order) == numCourses else []
```

```python
    left, right = 0, len(nums) - 1
    while left <= right:
        mid = left + (right - left) // 2  # 防溢出
```

```python
def permute(nums):
    result = []
    def backtrack(path, remaining):
        if not remaining:
            result.append(path[:])
            return
        for i in range(len(remaining)):
            path.append(remaining[i])
            backtrack(path, remaining[:i] + remaining[i+1:])
            path.pop()  # 撤销选择
    backtrack([], nums)
    return result

# 回溯：N 皇后
def solve_n_queens(n):
    result = []
    def backtrack(row, cols, diag1, diag2, board):
        if row == n:
            result.append(["".join(r) for r in board])
            return
        for col in range(n):
            if col in cols or row-col in diag1 or row+col in diag2:
                continue
            board[row][col] = 'Q'
            backtrack(row+1, cols|{col}, diag1|{row-col}, diag2|{row+col}, board)
            board[row][col] = '.'
    backtrack([], set(), set(), set(), [['.']*n for _ in range(n)])
    return result
```

### Day 1：Transformer 架构

**面试必答题**：

```
Q: "请简述 Transformer 的核心架构"

A: Transformer 由 Encoder-Decoder 组成：
1. Input Embedding + 位置编码（Positional Encoding）
2. Multi-Head Self-Attention：每个 token 关注序列中所有其他 token
   - Q = XW_q, K = XW_k, V = XW_v
   - Attention(Q,K,V) = softmax(QK^T / √d_k) × V
   - Multi-Head = 多组 (Q,K,V) 并行计算后拼接
3. Feed-Forward Network（两层 MLP）
4. 残差连接 + LayerNorm
5. Decoder 额外有 Cross-Attention（关注 Encoder 输出）

GPT = Decoder-only（自回归生成）
BERT = Encoder-only（双向理解）
T5 = Encoder-Decoder（seq2seq）
```

**追问**：
- Q: "为什么 Attention 要除以 √d_k？"
  - A: "防止点积值过大导致 softmax 梯度消失。当 d_k 很大时，QK^T 的方差约为 d_k，softmax 会将大部分概率集中在一个位置上，梯度接近 0。除以 √d_k 将方差归一化到 1。"

- Q: "位置编码为什么不用可学习的？"
  - A: "原始 Transformer 用三角函数位置编码是因为它能泛化到训练时没见过的序列长度。可学习的位置编码固定了最大长度。但现在 RoPE（旋转位置编码）更主流，它把位置信息编码到注意力计算中。"

### Day 4：状态管理与序列化

**你的 TaskResultStore 详解**：

```python
# 问题：LangGraph 的 TaskState 在节点间传递时会深拷贝
# 当 solver 输出几百 KB 的代码 + 结果时，深拷贝成为性能瓶颈

# 解决方案：State 外部化
class TaskResultStore:
    def __init__(self, base_dir: str):
        self.base_dir = Path(base_dir)
    
    def save(self, task_id: str, agent_name: str, data: dict) -> str:
        """保存 Agent 输出到磁盘，返回引用标记"""
        filepath = self.base_dir / task_id / f"{agent_name}_output.json"
        filepath.parent.mkdir(parents=True, exist_ok=True)
        
        with open(filepath, 'w') as f:
            json.dump(data, f)
        
        # state 中只存引用
        return f"__ref__:{filepath}"
    
    @staticmethod
    def load(ref: str) -> dict:
        """从引用标记加载数据"""
        if ref.startswith("__ref__:"):
            filepath = ref[8:]  # 去掉前缀
            with open(filepath, 'r') as f:
                return json.load(f)
        return ref  # 非引用，直接返回

# 效果：
# 之前：state = {"results": {"solver": {"code": "几千行...", "output": "几百KB..."}}}
#         → 深拷贝 MB 级数据
# 之后：state = {"results": {"solver": {"__ref__": "task_xxx/solver_output.json"}}}
#         → 深拷贝 KB 级引用

# 面试追问：
# Q: "磁盘读写不是比内存慢很多吗？"
# A: "是的，但只在需要时才读取（lazy loading）。大多数节点不需要
#     solver 的完整输出，只有 writer 需要。用 O(1) 的引用传递
#     替代 O(n) 的深拷贝，总体是值得的。"
```

### Day 1：BaseAgent + ReAct（base.py）

**你需要能手写的逻辑**：

```python
class BaseAgent:
    def __init__(self, name, model="sonnet"):
        self.name = name
        self.model = model
        self.llm_client = get_llm_client(model)
    
    async def run(self, context: dict) -> dict:
        """核心执行逻辑"""
        # 1. 构建系统提示词
        system_prompt = self._build_system_prompt()
        
        # 2. 构建用户消息（从 context 提取相关信息）
        user_message = self._build_user_message(context)
        
        # 3. 检查记忆
        relevant_memory = self.memory.retrieve_relevant(user_message)
        
        # 4. 组装最终 prompt
        messages = [
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": user_message}
        ]
        if relevant_memory:
            messages.insert(1, {"role": "system", "content": f"Relevant memory:\n{relevant_memory}"})
        
        # 5. 调用 LLM（支持 ReAct 工具循环）
        response = await self._react_loop(messages, tools=self._get_tools())
        
        # 6. 保存结果到记忆
        self.memory.save(user_message, response)
        
        return response
    
    async def _react_loop(self, messages, tools, max_iterations=5):
        """ReAct 工具循环"""
        for i in range(max_iterations):
            response = await self.llm_client.chat(
                messages=messages,
                tools=tools
            )
            
            if not response.tool_calls:
                return response.content  # 没有工具调用，返回最终结果
            
            # 执行工具调用
            for tool_call in response.tool_calls:
                result = await self._execute_tool(tool_call)
                messages.append({"role": "tool", "content": str(result)})
        
        return response.content  # 达到最大迭代次数
```

```python
import time
from enum import Enum

class CircuitState(Enum):
    CLOSED = "closed"       # 正常
    OPEN = "open"           # 熔断
    HALF_OPEN = "half_open" # 试探

class CircuitBreaker:
    def __init__(
        self,
        failure_threshold: int = 10,
        recovery_timeout: float = 300,
        window_size: float = 1800  # 30 分钟滑动窗口
    ):
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.window_size = window_size
        
        self.state = CircuitState.CLOSED
        self.failure_count = 0
        self.failure_timestamps = []  # 滑动窗口
        self.last_state_change = time.time()
    
    def _clean_old_failures(self):
        """清理窗口外的旧失败记录"""
        cutoff = time.time() - self.window_size
        self.failure_timestamps = [
            ts for ts in self.failure_timestamps if ts > cutoff
        ]
        self.failure_count = len(self.failure_timestamps)
    
    async def call(self, func, *args, **kwargs):
        """执行受熔断器保护的函数"""
        self._clean_old_failures()
        
        # 检查熔断状态
        if self.state == CircuitState.OPEN:
            elapsed = time.time() - self.last_state_change
            if elapsed > self.recovery_timeout:
                self.state = CircuitState.HALF_OPEN
                self.last_state_change = time.time()
            else:
                raise CircuitBreakerError(
                    f"Circuit OPEN, retry in {self.recovery_timeout - elapsed:.0f}s"
                )
        
        try:
            result = await func(*args, **kwargs)
            
            # 成功：如果是 HALF_OPEN，恢复到 CLOSED
            if self.state == CircuitState.HALF_OPEN:
                self.state = CircuitState.CLOSED
                self.failure_count = 0
                self.failure_timestamps = []
                self.last_state_change = time.time()
            
            return result
            
        except Exception as e:
            # 失败：记录失败
            self.failure_timestamps.append(time.time())
            self.failure_count += 1
            
            # 达到阈值：熔断
            if self.failure_count >= self.failure_threshold:
                self.state = CircuitState.OPEN
                self.last_state_change = time.time()
            
            raise

# 使用
```

```python
# 第 2 层：程序化校验
def _validate_no_fabrication(agent_output: dict) -> dict:
    flags = []
    score = 0.0
    
    # 检测无来源的价格/收益率
    content = agent_output.get("content", "")
    price_pattern = r'\$[\d,]+\.?\d*|¥[\d,]+\.?\d*'
    if re.search(price_pattern, content):
        if not agent_output.get("data_source"):
            flags.append("无来源价格数据")
            score += 0.3
    
    # 检测无引用的具体性能数字
    perf_pattern = r'accuracy|precision|recall|F1|AUC|R²'
    if re.search(perf_pattern, content, re.I):
        numbers = re.findall(r'(\d+\.\d+)%', content)
        for num in numbers:
            if float(num) > 100:
                flags.append(f"异常性能数字: {num}%")
                score += 0.5
    
    # 检测异常引用
    citation_pattern = r'\(([A-Z][a-z]+(?:\s(?:et\sal\.|&\s[A-Z][a-z]+))?),\s*(\d{4})\)'
    citations = re.findall(citation_pattern, content)
    current_year = datetime.now().year
    for author, year in citations:
        if int(year) > current_year:
            flags.append(f"未来引用: {author}, {year}")
            score += 0.4
    
    return {"flags": flags, "score": min(score, 1.0)}

# 第 3 层：数值范围校验
def validate_numerical_results(results: dict) -> List[str]:
    errors = []
    
    accuracy = results.get("accuracy")
    if accuracy is not None:
        if accuracy > 1.0:
            errors.append(f"Accuracy {accuracy} > 1.0")
        if accuracy < 0:
            errors.append(f"Accuracy {accuracy} < 0")
    
    r_squared = results.get("r_squared")
    if r_squared is not None:
        if r_squared > 1.0:
            errors.append(f"R² {r_squared} > 1.0")
    
    precision = results.get("precision")
    if precision is not None and precision < 0:
        errors.append(f"Precision {precision} < 0")
    
    return errors
```

```
     TokenBudgetManager 按类别预分配，关键字段永不压缩。
     如果需要更长的论文，可以升级到 Claude 的 200K context window。"

Q8: "如果 LLM API 返回空内容怎么办？"
A8: "三层检查：1) Adapter 层——检查 HTTP 状态码和响应格式；
     2) Agent 层——检查内容长度（<10 字符视为无效）；
     3) 编排器层——检查结果完整性（必填字段是否存在）。
     空内容会触发重试，3 次失败后熔断。"

Q9: "你的 checkpoint 机制和数据库事务有什么区别？"
A9: "checkpoint 是任务级的快照（每个 Agent 完成后保存），
     数据库事务是操作级的原子性（ACID）。
     checkpoint 允许部分完成（analyzer 完成但 solver 未完成），
     而事务要求全部完成或全部回滚。
     我用 JSON + 原子写入（.tmp + rename）实现轻量级持久化。"

Q10: "如果用户输入了一个系统无法处理的问题类型怎么办？"
A10: "preflight 节点会做问题分类。如果问题不在支持的模板范围内
     （数学建模/金融分析/CCF-A/综述），系统会返回提示信息，
     建议用户选择合适的模板或调整问题描述。"

Q11: "你的沙箱能防止恶意代码吗？"
A11: "能防止大部分学术场景的恶意代码。三层防御：
     resource limits 防止资源耗尽（CPU/内存），
     namespace 防止访问宿主机文件和网络，
     import restrictions 防止调用危险模块。
     但无法防止所有攻击（如内核漏洞利用），
     对于不可信代码建议用 Docker 或 gVisor。"

Q12: "为什么用 TF-IDF 而不是向量数据库？"
A12: "三个原因：1) 部署简单——零外部依赖，pip install 即可；
     2) 规模匹配——本项目知识库是几百到几千篇论文，TF-IDF 够用；
     3) 成本为零——不需要 GPU 推理嵌入向量。
     如果未来需要语义检索，可以升级到 ChromaDB + sentence-transformers。"

Q13: "你的系统和 MetaGPT 有什么区别？"
A13: "核心区别：1) MetaGPT 是通用多 Agent 框架，我的系统专注论文生成；
     2) MetaGPT 没有 LangGraph 的 checkpoint/resume；
     3) 我的系统有 Anti-Fabrication 防编造机制；
     4) 我的系统有三级记忆 + 跨任务学习。
     MetaGPT 更适合做通用软件开发，我的系统更适合学术论文。"

Q14: "如果要支持 100 个 Agent 并发，你会怎么改？"
A14: "三个改动：1) TaskResultStore 从 JSON 迁移到 Redis/PostgreSQL；
     2) LangGraph 从单机迁移到分布式（如 Ray）；
     3) LLM 调用从同步改为异步队列（Celery/RQ）。
     当前架构已经支持异步，扩展性较好。"

Q15: "你的系统最薄弱的环节是什么？"
A15: "引用准确性。虽然验证了 arXiv ID 存在，但无法验证引用内容是否
     准确对应原文。这是 LLM 幻觉的固有问题，目前没有完美解决方案。
     我的方案是在 peer_review 阶段增加引用验证，
     但最终仍需人工审核。"

Q16: "如果 LLM 的价格涨了 10 倍，你的系统还能用吗？"
A16: "能，但需要调整。三个优化：1) 用更便宜的模型（DeepSeek-V3 替代 Sonnet）；
     2) 减少 Agent 调用次数（合并小 Agent）；
     3) 用本地模型（Ollama + LLaMA）替代云端 API。
     系统已经支持 15+ Provider，切换成本很低。"

Q17: "你怎么保证 Agent 之间的消息不丢失？"
A17: "两层保证：1) TaskState 是内存中的 dict，LangGraph 节点间传递不经过网络；
     2) 关键输出通过 TaskResultStore 持久化到磁盘。
     即使进程崩溃，checkpoint 也能恢复到最后一个完成的 Agent。"

Q18: "你的系统有没有做 A/B 测试？"
A18: "有评测体系但没有正式的 A/B 测试框架。过程评测（代码成功率、
     熔断率）和结果评测（LaTeX 编译、同行评议评分）可以对比不同配置的效果。
     比如切换 LLM Provider 后，对比论文质量评分。
     未来可以加 formal A/B 测试。"

Q19: "如果面试官说'你的项目看起来像 AI 帮你写的'，你怎么回应？"
A19: "坦诚承认用了 AI 辅助工具，但强调：
     1) 架构设计和核心决策是我做的；
     2) 我能解释每一行代码的原理和设计动机；
     3) 解决的 6 个工程难题是基于真实调试经验；
     4) AI 辅助不等于不需要理解——相反，理解代码才能正确使用 AI。"

Q20: "你下一步想改进什么？"
A20: "三个方向：1) 引入 Docker 沙箱替代 namespace（更强安全性）；
     2) 集成 ChromaDB 替代 TF-IDF（更高质量检索）；
     3) 建立自动化论文质量评测框架（与已发表论文对比）。
     长期来看，想加入 RLHF 机制让系统从用户反馈中学习。"
```

### Day 1：网络

```
必须掌握：

1. TCP 三次握手/四次挥手
   客户端 → SYN → 服务端
   服务端 → SYN+ACK → 客户端
   客户端 → ACK → 服务端（连接建立）
   
   客户端 → FIN → 服务端
   服务端 → ACK → 客户端
   服务端 → FIN → 客户端
   客户端 → ACK → 服务端（连接关闭）

2. HTTP vs HTTPS vs WebSocket vs SSE
   HTTP：请求-响应，无状态
   HTTPS：HTTP + TLS 加密
   WebSocket：全双工，长连接（你的 ChatRoom 可以用）
   SSE：服务端单向推送，基于 HTTP（你的项目用这个）

3. TCP vs UDP
   TCP：可靠、有序、面向连接
   UDP：不可靠、无序、无连接

面试追问：
Q: "SSE 和 WebSocket 的区别？"
A: "SSE 是服务端单向推送（基于 HTTP），WebSocket 是全双工（独立协议）。
    SSE 更简单（不需要额外协议升级），适合服务端推送场景。
    WebSocket 适合双向通信（如在线协作编辑器）。
    本项目用 SSE 是因为 Agent 消息是单向推送，不需要客户端频繁发送。"
```

### Day 2：操作系统

```
必须掌握：

1. 进程 vs 线程 vs 协程
   进程：资源分配单位，独立内存空间
   线程：CPU 调度单位，共享进程内存
   协程：用户态线程，由程序调度（Python asyncio）

2. 死锁的四个必要条件 + 预防
   互斥、持有并等待、不可剥夺、循环等待
   预防：破坏任一条件（如资源有序分配）

3. 虚拟内存 + 页面置换
   页表、TLB、缺页中断
   LRU/FIFO/Clock 置换算法

4. 用户态 vs 内核态
   系统调用触发用户态→内核态切换
   你的沙箱用的 resource.setrlimit 是系统调用

面试追问：
Q: "Python 的 GIL 是什么？为什么 asyncio 不受 GIL 影响？"
A: "GIL（全局解释器锁）确保同一时刻只有一个线程执行 Python 字节码。
    asyncio 是单线程的，不存在并发竞争，所以 GIL 不影响。
    但如果用 asyncio 调用了同步阻塞代码（如 time.sleep），
    会阻塞整个事件循环，其他协程也无法执行。"
```

# 第九部分：项目改进与诚实面试（第 7-8 周补充）

## 9.1 代码审查发现的真实问题

**面试中如果被问"你的项目有什么不足"，以下是基于代码审查的诚实回答**：

### 问题 1：NAS/Loss Design 评估器是假的

```
现状：nas.py 的 _default_evaluator 不实际训练模型，
     只基于参数量估算返回 0.3/0.5/0.7 假分数。
     loss_design.py 的评估器基于表达式深度返回假分数。

面试回答：
"NAS 和 Loss Design 的评估器目前是占位实现——用启发式规则返回假分数，
 而不是实际训练模型。原因是单卡 8GB 显存限制，
 实际训练一个 NAS 搜索需要的计算量远超可用资源。

 改进方案：接入 GPUExecutor，在 CIFAR-10 上用 5 epoch 快速评估。
 但由于显存限制，种群大小只能设为 10，代数设为 5，
 搜索深度有限。

 这个模块的定位是'框架就绪，评估器待接入真实 GPU 集群'。"
```

### 问题 2：两个沙箱实现未统一

```
现状：services/code_sandbox.py（静态扫描 + resource limits）
     和 core/sandbox.py（namespace 隔离 + import hook）
     是两套独立实现，API 不同。

面试回答：
"项目中有两个沙箱实现，这是一个技术债务。
 核心 Agent（solver/data）用 core/sandbox.py（有 namespace 隔离），
 实验执行用 services/code_sandbox.py（只有 resource limits）。

 原因是两个模块由不同时期开发，没有统一规划。
 改进方案是合并为一个 CodeSandbox，通过配置开关控制 namespace/import hook。
 这是 P0 优先级的改进项。"
```

### 问题 3：网络隔离是假的

```
现状：code_sandbox.py 的网络隔离只是设置
     HTTP_PROXY=http://127.0.0.1:0，任何忽略代理的代码可绕过。

面试回答：
"网络隔离目前是环境变量级别的，不是真正的 namespace 隔离。
 代码注释中明确标注了'placeholder only, actual blocking requires iptables'。

 对于学术代码执行场景（numpy/pandas/matplotlib）够用，
 因为这些库不会绕过代理环境变量。
 但如果要处理不可信代码，需要用 Linux namespace 或 Docker。"
```

### 问题 4：Token 估算粗糙

```
现状：context_compressor.py 用 CHARS_PER_TOKEN = 3，
     中英文混合时误差可达 2x。

面试回答：
"Token 估算用的是字符数除以 3 的粗略方法，
 中文 1 个字约 2-3 tokens，英文 1 个单词约 1-1.5 tokens，
 混合文本误差可达 2 倍。

 改进方案是接入 tiktoken（OpenAI 的分词器）做精确计数。
 但 tiktoken 只支持 OpenAI 模型，对于 Anthropic/Gemini 模型
 需要维护多个分词器。当前方案是'够用但不精确'。"
```

### 问题 5：Episodic Memory 压缩是规则基础

```
现状：memory.py 的 compress() 做简单字符串拼接，
     docstring 明确标注'生产环境应使用 LLM 智能摘要'。

面试回答：
"Episodic Memory 的压缩目前是规则基础——按 Agent 分组后拼接字符串。
 这是一个已知的技术债务，docstring 中标注了'生产环境应使用 LLM'。

 选择规则基础的原因是：1) 延迟——LLM 摘要需要额外调用；
 2) 成本——每次压缩都调 LLM 不划算。
 但规则基础的压缩质量确实不如 LLM 智能摘要。
 改进方案是'低频调用 LLM + 高频用规则'的混合策略。"
```

## A.2 异步迭代器

```python
class AgentPipeline:
    def __init__(self, agents):
        self.agents = agents
    
    def __aiter__(self):
        self.index = 0
        return self
    
    async def __anext__(self):
        if self.index >= len(self.agents):
            raise StopAsyncIteration
        agent = self.agents[self.index]
        self.index += 1
        return agent

# 使用
async for agent in AgentPipeline(agents):
    result = await agent.run(context)
```

## A.3 描述符

```python
class ValidatedField:
    def __init__(self, min_value=None, max_value=None):
        self.min_value = min_value
        self.max_value = max_value
    
    def __set_name__(self, owner, name):
        self.name = name
    
    def __get__(self, obj, objtype=None):
        if obj is None:
            return self
        return obj.__dict__.get(self.name)
    
    def __set__(self, obj, value):
        if self.min_value is not None and value < self.min_value:
            raise ValueError(f"{self.name} must be >= {self.min_value}")
        if self.max_value is not None and value > self.max_value:
            raise ValueError(f"{self.name} must be <= {self.max_value}")
        obj.__dict__[self.name] = value

class AgentConfig:
    temperature = ValidatedField(min_value=0.0, max_value=2.0)
    max_tokens = ValidatedField(min_value=1, max_value=100000)
```

## A.4 元类

```python
class AgentMeta(type):
    """自动注册所有 Agent 子类"""
    _registry = {}
    
    def __init__(cls, name, bases, attrs):
        super().__init__(name, bases, attrs)
        if name != "BaseAgent":  # 排除基类本身
            AgentMeta._registry[name] = cls
    
    @classmethod
    def get_agent(mcs, name):
        return mcs._registry.get(name)

class BaseAgent(metaclass=AgentMeta):
    pass

class AnalyzerAgent(BaseAgent):
    pass

class SolverAgent(BaseAgent):
    pass

# 自动注册
print(AgentMeta._registry)
# {'AnalyzerAgent': <class 'AnalyzerAgent'>, 'SolverAgent': <class 'SolverAgent'>}
```

# 附录 E：v7.0 新特性深度拆解

## E.1 需求分解器（RequirementDecomposer）

**设计背景**：用户输入万字以上长提示词时，直接传给 LLM 会导致 token 超限或理解不完整。

**实现方案**：
```
用户输入(>3000字) → RequirementDecomposer Agent → 结构化 JSON 计划
  ↓
所有 Agent 通过 context["requirement_plan"] 读取
```

**输出 Schema**：
```json
{
  "research_goal": "一句话研究目标",
  "background": "研究背景摘要",
  "key_questions": ["核心问题1", "核心问题2"],
  "methodology_hints": ["方法提示1", "方法提示2"],
  "expected_output": "预期产出类型",
  "data_requirements": ["需要的数据类型"],
  "template_suggestion": "推荐模板ID",
  "subtasks": [
    {"id": 1, "description": "...", "suggested_agent": "research_agent", "priority": "high"}
  ]
}
```

**面试追问**：
- Q: "为什么用 3000 字作为阈值？"
  - A: "经验值。3000 字约等于 4000-5000 token，超过这个长度 LLM 对细节的注意力会下降。拆解后每个子任务的 context 更聚焦，LLM 输出质量更高。"

- Q: "拆解后的计划存储在哪里？所有 Agent 都能读取吗？"
  - A: "存储在项目目录的 requirement_plan.json。通过 LangGraph 的 TaskState 注入到每个 Agent 的 context 中，所有 Agent 都可以读取。"

## E.2 创新发现 Agent（InnovationAgent）

**设计背景**：research_agent 返回文献后，需要有人系统性地分析研究空白并提出创新方案。

**工作流程**：
```
research_agent 输出(papers, methods) + analyzer 输出(problem_type, gaps)
  ↓
InnovationAgent 分析
  ↓
输出：research_gaps + innovation_ideas + recommended_approach
```

**创新方案评分维度**：
- novelty（新颖性）：与现有工作的差异度
- feasibility（可行性）：技术实现难度
- expected_contribution（预期贡献）：学术价值

**面试追问**：
- Q: "创新发现和 research_agent 有什么区别？"
  - A: "research_agent 负责'找文献'，innovation_agent 负责'找空白'。research 输出是'别人做了什么'，innovation 输出是'别人没做什么、我们可以做什么'。"

- Q: "创新方案的质量怎么保证？"
  - A: "三重验证：1) LLM 自评新颖性；2) 与 research_agent 的论文列表交叉验证（不能重复已有的）；3) feasibility 评估过滤掉不切实际的方案。"

## E.3 知识库自动分类（KnowledgeOrganizer）

**四维度分类体系**：

| 维度 | 分类 | 规则示例 |
|------|------|----------|
| 类型 | paper/dataset/code/benchmark | .pdf→paper, .csv→dataset, .py→code |
| 领域 | NLP/CV/RL/optimization | 文件名关键词匹配 |
| 方法 | transformer/CNN/GNN/RL | 内容摘要关键词 |
| 质量 | top_venue/good/average | arXiv ID + venue 匹配 |

**实现方式**：规则优先 + LLM 增强（基础分类无需 LLM，精确分类调用轻量模型）

**面试追问**：
- Q: "为什么不直接用 LLM 分类？"
  - A: "成本和延迟。大部分资源可以通过文件扩展名和文件名规则快速分类，只有边界case才需要 LLM。比如 .pdf 一定是 paper，.py 一定是 code，不需要浪费 token。"

## E.4 任务总结 Agent（SummaryAgent）

**总结报告结构**：
```
task_summary = {
  "research_summary": "研究过程回顾",
  "innovation_points": ["创新点1", "创新点2"],
  "experiment_results": {"key_findings": [...], "metrics": {...}},
  "paper_quality": {"overall_score": 75, "strengths": [...], "weaknesses": [...]},
  "lessons_learned": [
    {"category": "method_selection", "content": "...", "tags": ["tag1"], "impact_score": 7}
  ],
  "dataset_registry": [{"name": "...", "source": "..."}],
  "literature_registry": [{"title": "...", "arxiv_id": "..."}],
  "recommendations": ["建议1", "建议2"]
}
```

**经验分类体系**：
```
method_selection | data_processing | writing | experiment_design | template_specific
```

**use_count 闭环**：经验被检索引用时自动 +1，使用越多的经验排序越靠前。

**面试追问**：
- Q: "经验提取的 use_count 闭环是怎么工作的？"
  - A: "retrieve_relevant() 检索相关经验时自动 increment_use()，use_count 越高的经验在排序中权重越大。这样'被验证过多次'的经验会被优先复用，形成正反馈循环。"

## E.5 实验自动迭代

**质量评估标准**：
- 是否包含 baseline 对比实验
- 是否有 ablation study
- 实验是否成功执行

**迭代流程**：
```
experimentation_agent 执行实验
  ↓
评估质量（_evaluate_experiment_quality）
  ↓ 不满足
生成反馈（_generate_iteration_feedback）
  ↓
experimentation_agent 重新执行（最多 3 轮）
  ↓ 满足或达到上限
进入写作阶段
```

**面试追问**：
- Q: "实验迭代和 peer_review 修订有什么区别？"
  - A: "实验迭代是'实验质量不足时自动补全'（缺少 baseline 就加 baseline），peer_review 修订是'论文写作质量不足时重写'（评分低就改论文）。前者改实验，后者改论文。"

# 附录 F：安全审计相关面试题

## F.1 沙箱安全

```
Q: "你的沙箱之前禁用了导入拦截，为什么？怎么修复的？"

A: "v5.0-v5.2 禁用 Layer 3 是因为 numpy/pandas 在初始化时会导入 os/sys/subprocess，
    过于严格的导入限制会破坏科学计算库。v7.0 重新启用了导入拦截，改用
    sys._getframe(1) 检查直接调用者：如果调用者在 site-packages 中（库内部传递性
    导入），放行；如果调用者在用户代码中（直接导入危险模块），阻止。

    这样既保护了用户代码不能直接 import socket/urllib/http，又允许 numpy 内部
    导入 subprocess（由 RLIMIT_NPROC=0 阻止 fork）。"
```

```
Q: "命令执行白名单怎么设计的？"

A: "environments/run 端点只允许 python/pip/conda/node/latexmk 等安全命令。
    阻止 shell 元字符（;|&|$`）防止命令注入。
    cwd 参数限制在 /tmp、/home、项目目录内，防止路径穿越。
    requirements_path 同样限制路径范围。"
```

```
Q: "API 认证为什么选 API Key 而不是 JWT？"

A: "这是一个本地研究工具，不是公网服务。API Key 方案：
    1) 实现简单：一个环境变量 MATHMODEL_API_KEY + 一个 FastAPI 依赖
    2) 无状态：不需要 session 管理、token 刷新
    3) 够用：保护危险端点（debug、environments/run、settings POST）
    
    如果是公网部署，JWT + RBAC 是更好的选择。"
```

## F.2 SSRF 防护

```
Q: "Provider 测试端点的 SSRF 风险怎么处理？"

A: "POST /providers/{id}/test 向用户指定的 api_host 发送请求。
    风险：攻击者设置 api_host=http://169.254.169.254 访问云元数据。
    
    防护方案：
    1) api_host 必须是已注册 Provider 的 api_host（不接受任意 URL）
    2) 屏蔽私网 IP（10.x, 172.16-31.x, 192.168.x, 169.254.x）
    3) 仅允许 http/https 协议"
```

## F.3 Prompt 注入防御

```
Q: "用户输入的 problem_text 直接嵌入 LLM prompt，怎么防注入？"

A: "三层防御：
    1) 结构化分隔：system prompt 用 XML 标签包裹用户输入
       <user_input>{problem_text}</user_input>
       模型不容易将用户内容误解为指令
    2) 输出校验：_validate_no_fabrication 检测异常输出
    3) 沙箱隔离：即使 LLM 被注入执行了危险代码，也被沙箱阻止"
```

## G.1 Next.js 14

```
Q: "为什么选 Next.js 而不是纯 React？"

A: "三个原因：

    2) App Router：React Server Components 减少客户端 JS 体积
    3) 内置优化：图片优化、字体优化、脚本加载策略
    
    本项目主要用客户端渲染（CSR），因为是内部工具不需要 SEO。
```

```
Q: "React Server Components 和 Client Components 的区别？"

A: "Server Components 在服务端渲染，发送 HTML 到客户端，不能用 useState/useEffect。
    Client Components 在客户端渲染，可以交互。
    
    本项目的 page.tsx 是 Client Components（用 'use client'），
    因为需要 useState 管理状态、useEffect 调 API。"
```

## G.2 Zustand 状态管理

```
Q: "为什么选 Zustand 而不是 Redux？"

A: "三个原因：
    1) 轻量：Zustand ~1KB vs Redux+RTK ~10KB
    2) 简单：没有 action/reducer 概念，直接 set()
    3) 够用：本项目状态简单（selectedFiles, activeProject），
       不需要 Redux 的中间件、devtools 等复杂功能
    
    Zustand 的 create() 一行代码定义 store：
    const useStore = create((set) => ({
      selectedFiles: [],
      setSelectedFiles: (files) => set({ selectedFiles: files }),
    }))"
```

## G.3 SSE 实时推送

```
Q: "前端怎么接收 Agent 的实时消息？"

A: "使用 EventSource（SSE）而非 WebSocket：
    1) SSE 是单向推送（服务端→客户端），适合 Agent 输出流
    2) WebSocket 是双向通信，本项目不需要客户端实时发消息给 Agent
    3) SSE 自动重连，WebSocket 需要手动处理
    
    实现：
    const es = new EventSource('/api/v1/tasks/{id}/stream');
    es.addEventListener('phase_changed', (evt) => {
      const data = JSON.parse(evt.data);
      setMessages(prev => [...prev, data]);
    });"
```

# 附录 H：RAG 高级技术

## H.1 HyDE（假设文档嵌入）

```
原理：
  普通 RAG：query → embedding → 检索 → 结果
  HyDE：query → LLM 生成假设文档 → embedding → 检索 → 结果
  
  优势：假设文档与目标文档在语义空间更接近，检索质量更高

你的项目关联：
  research_agent 搜索文献时，可以先让 LLM 生成"理想的论文摘要"，
  再用这个摘要去检索，比直接用用户问题检索更精准。
```

## H.2 多模态 RAG

```
挑战：PDF 中的图表、公式无法用文本 embedding 检索
  
解决方案：
  1) 图表：用 LLM 描述图表内容，生成文本摘要后 embedding
  2) 公式：LaTeX 转文本描述（如"二次规划问题"）
  3) 表格：转为结构化文本（如"方法A准确率95%，方法B准确率92%"）

你的项目关联：
  data_agent 的 OCR 功能（调 LLM 识别图片文字）就是多模态处理的雏形。
```

## H.3 Query 改写策略

```
1. HyDE：生成假设文档再检索
2. Multi-Query：一个问题生成多个子问题，分别检索后合并
3. Step-Back：先生成高层问题再检索（如"什么是attention"→"attention的数学公式"）
4. Sub-Query Decomposition：复杂问题拆分为简单子问题

你的项目关联：
  requirement_decomposer 本质上就是 Sub-Query Decomposition。
  万字长提示词拆解为子任务，每个子任务独立检索和执行。
```
