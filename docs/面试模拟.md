## 第一部分：项目介绍模拟（5 分钟）

### 开场白

> 面试官您好，我介绍一下我最近做的项目。这是一个**多智能体协作论文生产系统**，目标是实现从问题分析到论文产出的全流程自动化。
>
> 核心架构是 **LangGraph 状态机编排 + 15 个 Agent 协作**。LangGraph 相比 LangChain Agent 的优势在于：显式的状态管理（TypedDict）、条件边路由、原生 checkpoint 断点续传。
>
> 项目最大的技术亮点是**零幻觉架构**——5 层防御体系确保输出的真实性：
> 1. **AST 代码审计**：在代码执行前，用 Python AST 解析检测硬编码指标（如 `accuracy = 0.95`）
> 2. **沙箱执行**：4 层安全防御（资源限制、网络隔离、导入拦截、静态扫描）
> 3. **符号审计**：用确定性算法验证表格加总、百分比、指标范围
> 4. **事实核查**：LaTeX 数字 vs 求解器实际输出交叉验证
> 5. **参考文献验真**：DOI/arXiv ID 通过 CrossRef/arXiv API 验证存在性
>
> 这套架构的核心思想是 **Code-as-Truth**：LLM 只生成代码，数字由沙箱真实运行产生，杜绝编造。

## 第二部分：技术深度问答

### Q1: 为什么选择 LangGraph 而不是 LangChain Agent？

**考察点**：对编排框架的理解深度

**参考答案**：

```
LangChain Agent 是单一的 while 循环，隐式状态管理，不支持断点续传。
LangGraph 是 StateGraph + TypedDict，显式状态管理，支持条件边、并行、checkpoint。

关键差异：
1. 状态可见性：LangGraph 的 TaskState 是 TypedDict，所有字段类型明确；
   LangChain Agent 的 state 是隐式的，容易出 bug
2. 流程控制：LangGraph 支持条件边（if/else 路由）、循环（自动迭代）；
   LangChain Agent 只能线性执行
3. 断点续传：LangGraph 原生 checkpoint，任务中断可从任意节点恢复；
   LangChain Agent 不支持
4. 可调试性：LangGraph 每个节点独立，可单独测试；
   LangChain Agent 是黑盒

我们的场景需要 15 个 Agent 协作、条件分支、自动迭代、断点续传，
LangGraph 是唯一合理选择。
```

### Q2: 你的多 Agent 之间如何通信？

**考察点**：分布式系统设计能力

**参考答案**：

```
我们采用"黑板模式 + 消息总线"的混合架构：

1. 同步状态共享（黑板模式）：
   - 所有 Agent 通过 TaskState 读写共享状态
   - Agent 从 state 读取输入，执行后写入 results[agent_name]
   - 好处：简单直接，无通信开销

2. 异步消息总线（ChatRoom）：
   - Agent 讨论时互相发送消息
   - 支持 SSE 实时推送到前端
   - 用户可实时介入讨论
   - 好处：支持人类参与，可追溯

3. 输出归一化：
   - 三种建模 Agent（modeler/algorithm_engineer/financial_analyst）
   - 输出统一为 sub_problem_models 标准格式
   - solver 和 writer 不需要感知具体建模 Agent

为什么不用消息队列（RabbitMQ/Kafka）：
- 这是单机系统，Agent 数量固定（15 个）
- 消息队列引入额外复杂度，但收益不大
- TaskState 已经足够满足通信需求
```

### Q3: ReAct 循环的 Token 预算是怎么管理的？

**考察点**：LLM 调用优化能力

**参考答案**：

```
LLM 上下文窗口有限，ReAct 循环容易超限。我们用三层策略管理：

1. 动态迭代次数：
   - 基于工具数量和上下文长度预估：len(tools) * 2 * (context_length / 8000) * 1.5
   - 硬上限 20 次，防止无限循环

2. 滑动窗口 + 摘要混合（参考 MemGPT）：
   - 保留最近 6 轮完整历史
   - 旧的 tool call 压缩为摘要
   - 例：第 7 轮之前的历史合并为一段文字描述

3. Token 预算分配：
   - System Prompt: 5%
   - User Context: 20%
   - ReAct History: 15%
   - Summary Buffer: 5%
   - Tool Descriptions: 5%
   - Reserve: 50%

4. 实时监控：
   - 每 60 秒检查 token 使用量
   - 超过预算 80% 自动触发压缩
```

### Q4: 你的 RAG 系统是怎么做的？

**考察点**：检索增强生成能力

**参考答案**：

```
我们用混合检索（语义 + BM25）+ RRF 融合：

1. 语义检索：
   - sentence-transformers/all-MiniLM-L6-v2 生成 embedding
   - 余弦相似度检索
   - 适合语义相近但关键词不同的查询

2. BM25 关键词检索：
   - 基于词频和逆文档频率
   - 适合精确关键词匹配
   - 使用 rank_bm25 库

3. RRF（Reciprocal Rank Fusion）融合：
   - score(d) = 1/(k + rank_semantic(d)) + 1/(k + rank_bm25(d))
   - k=60 是经验参数
   - 无需归一化，直接按分数排序

4. 作用域隔离：
   - Global KB：所有项目共享
   - Project KB：仅指定项目可见
   - 自动注入优先级：显式选择 > 项目 KB > 全局 KB

为什么不用向量数据库（Milvus/Pinecone）：
- 项目规模较小（几百篇文档）
- sentence-transformers 本地推理足够快
- 减少外部依赖
```

### Q5: AST 代码审计的原理是什么？

**考察点**：静态分析能力

**参考答案**：

```
AST（抽象语法树）是 Python 代码的树形结构表示。
我们用 ast 模块解析代码，然后用 Visitor 模式遍历树节点：

1. 硬编码指标检测：
   - 匹配 Assign 节点：targets 是 Name，name 在 HARDCODED_METRICS 集合中
   - 检查 value 是否为 Constant/Num（常量）
   - 例：accuracy = 0.95 → 发现硬编码

2. 伪造输出检测：
   - 匹配 Expr 节点：func 是 print
   - 检查参数是否包含固定数字模式
   - 例：print("Accuracy: 95.2%") → 发现伪造输出

3. 数据源检测：
   - 检查是否有 read_csv/read_excel 等函数调用
   - 没有则警告缺少数据源

4. 内联数据检测：
   - 检查列表字面量中是否包含大量数字常量（>20 个）
   - 有则警告疑似硬编码数据集

评分：100 - (20 × error_count) - (5 × warning_count)
有 error 则不通过，warning 不阻断执行。
```

### Q6: 沙箱的四层防御具体是什么？

**考察点**：系统安全能力

**参考答案**：

```
Layer 1: 静态分析
- 正则扫描危险调用：os.system、subprocess.call、eval、exec
- 发现则拒绝执行

Layer 2: 资源限制
- Python resource.setrlimit 设置：
  - CPU 时间：120 秒
  - 内存：1GB
  - 文件大小：100MB
- 超限自动 kill

Layer 3: 网络隔离
- unshare --net 创建新网络命名空间
- 代码无法访问外部网络
- 防止数据泄露和恶意下载

Layer 4: 执行隔离
- subprocess.Popen 在独立进程执行
- stdout/stderr 管道捕获
- timeout 控制超时

另外还有导入拦截（Layer 3.5）：
- import hook 拦截 os、subprocess、socket 等危险模块
- 通过 sys.meta_path 注入自定义 finder
```

### Q8: 你的记忆系统是怎么设计的？

**考察点**：长期记忆能力

**参考答案**：

```
三级架构：

1. Working Memory（任务上下文）：
   - 当前 TaskState
   - Agent 私有记忆池（关键词检索）
   - 全局论文记忆池（术语/符号/结论一致性）

2. Episodic Memory（任务级）：
   - Agent 执行日志
   - 讨论记录
   - 时间线回放（debug-history API）

3. Lessons Memory（跨任务）：
   - 任务完成后自动提取经验
   - 标签检索 + 影响力评分
   - 用户反馈更新 use_count
   - 存储：backend/data/memory/lessons.json

全局论文记忆池解决长论文的术语不一致问题：
- terminology：术语表（确保全文使用统一术语）
- symbols：符号表（确保符号含义一致）
- key_claims：关键结论（确保后续章节引用正确）
```

### Q9: GPU 显存监控和 OOM 防护是怎么做的？

**考察点**：GPU 编程能力

**参考答案**：

```
VRAMMonitor 实时监控：

1. 监控策略：
   - 独立线程，2 秒间隔
   - 读取 torch.cuda.memory_allocated()
   - 计算使用率：usage = allocated / total_memory

2. 告警阈值：
   - 85%：warning 日志
   - 95%：触发 OOM 保护

3. OOM 保护：
   - 向训练进程发送 SIGTERM
   - 训练进程收到信号后保存 checkpoint
   - 等待 5 秒后 SIGKILL

4. 显存估算：
   - 训练前估算推荐 batch size
   - 估算公式：batch_size = available_memory / (model_params * 4 * 3)
   - 4 bytes per parameter, 3x for gradients + optimizer states

CheckpointManager：
- 每 N 个 epoch 自动保存
- 保留最近 3 个 checkpoint
- 支持从最新 checkpoint 恢复训练
```

### Q10: 你是怎么处理 LLM 的 hallucination（幻觉）问题的？

**考察点**：LLM 局限性理解

**参考答案**：

```
我们的核心思想是 Code-as-Truth：LLM 只生成代码，数字由真实执行产生。

但 LLM 的幻觉还体现在：
1. 编造不存在的参考文献 → 参考文献验真（CrossRef/arXiv API）

5 层防御：
LLM 生成 → AST 审计 → 沙箱执行 → 符号审计 → 事实核查

每一层都有明确的检测目标和通过标准，
任何一层失败都会标记问题并通知用户。
```

### Q11: 你的防死亡螺旋机制是怎么设计的？

**考察点**：系统鲁棒性设计

**参考答案**：

```
死亡螺旋是指：代码执行失败 → 重试 → 再失败 → 再重试的循环，
浪费计算资源且无法推进流程。

我设计了三机制防护：

1. 组件化注入（restricted 模式）：
   - Coder 只生成 nn.Module/Loss 组件
   - 系统自动注入到预置的 Base Template
   - 降低代码出错概率

2. AST 安全壳：
   - SafetyShellTransformer 自动变换 AST
   - 包裹 try-except 防止未捕获异常
   - 注入 cuda.empty_cache() 防 OOM
   - 注入 gc.collect() 释放内存

3. 渐进式越狱熔断：
   - 监控 error_count 和指标趋势
   - error_count >= 3 → 降级为 restricted（死亡螺旋熔断）
   - 指标连续未提升 → 升级为 jailbreak（模板瓶颈越狱）
   - 熔断阈值动态调整（3 → 1）

所有经过 iterative_solver 的模板都接入保护，
CCF-A 模板额外启用组件化注入和越狱熔断。
```

### 题目 1：实现一个简单的 ReAct Agent

```python
# 考察：ReAct 循环、工具调用、错误处理

import json
from typing import List, Dict, Callable

class SimpleReActAgent:
    def __init__(self, tools: Dict[str, Callable], max_iterations: int = 5):
        self.tools = tools
        self.max_iterations = max_iterations

    def run(self, query: str, llm_call: Callable) -> str:
        """
        ReAct 循环：
        1. 用 LLM 生成 Thought + Action
        2. 执行 Action，得到 Observation
        3. 重复直到 LLM 输出 Final Answer
        """
        history = [{"role": "user", "content": query}]

        for i in range(self.max_iterations):
            # 1. LLM 生成 Thought + Action
            prompt = self._build_prompt(history)
            response = llm_call(prompt)

            # 2. 解析响应
            parsed = self._parse_response(response)

            if "final_answer" in parsed:
                return parsed["final_answer"]

            # 3. 执行 Action
            action = parsed["action"]
            action_input = parsed["action_input"]

            if action not in self.tools:
                observation = f"Error: Unknown tool {action}"
            else:
                try:
                    observation = self.tools[action](action_input)
                except Exception as e:
                    observation = f"Error: {e}"

            # 4. 更新历史
            history.append({"role": "assistant", "content": response})
            history.append({"role": "user", "content": f"Observation: {observation}"})

        return "达到最大迭代次数，未找到答案"

    def _build_prompt(self, history: List[Dict]) -> str:
        tools_desc = "\n".join([f"- {name}: {func.__doc__}" for name, func in self.tools.items()])
        return f"""你是一个 ReAct Agent。可用工具：
{tools_desc}

格式：
Thought: <你的思考>
Action: <工具名>
Action Input: <输入>

或：
Thought: <你的思考>
Final Answer: <最终答案>

历史：
{json.dumps(history, ensure_ascii=False)}"""

    def _parse_response(self, response: str) -> Dict:
        if "Final Answer:" in response:
            return {"final_answer": response.split("Final Answer:")[-1].strip()}
        lines = response.strip().split("\n")
        action = ""
        action_input = ""
        for line in lines:
            if line.startswith("Action:"):
                action = line.split(":", 1)[1].strip()
            elif line.startswith("Action Input:"):
                action_input = line.split(":", 1)[1].strip()
        return {"action": action, "action_input": action_input}
```

### 题目 2：实现一个简单的 RRF 融合

```python
# 考察：排序融合算法

from typing import List, Dict, Tuple

def reciprocal_rank_fusion(
    results_a: List[Dict],
    results_b: List[Dict],
    k: int = 60
) -> List[Tuple[str, float]]:
    """
    Reciprocal Rank Fusion

    score(d) = 1/(k + rank_a(d)) + 1/(k + rank_b(d))

    Args:
        results_a: 第一个检索结果列表 [{"id": "...", "score": ...}, ...]
        results_b: 第二个检索结果列表
        k: 平滑参数（默认 60）

    Returns:
        融合后的排序结果 [(doc_id, score), ...]
    """
    scores: Dict[str, float] = {}

    # 为每个结果计算 RRF 分数
    for rank, doc in enumerate(results_a):
        doc_id = doc["id"]
        scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank + 1)

    for rank, doc in enumerate(results_b):
        doc_id = doc["id"]
        scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank + 1)

    # 按分数降序排序
    sorted_results = sorted(scores.items(), key=lambda x: x[1], reverse=True)

    return sorted_results
```

### 题目 3：实现一个令牌桶限流器

```python
# 考察：并发控制、异步编程

import asyncio
import time

class AsyncTokenBucket:
    """异步令牌桶限流器"""

    def __init__(self, rate: float, capacity: int):
        """
        Args:
            rate: 每秒产生的 token 数
            capacity: 桶的最大容量
        """
        self.rate = rate
        self.capacity = capacity
        self.tokens = capacity
        self.last_refill = time.time()
        self._lock = asyncio.Lock()

    async def acquire(self):
        """获取一个令牌，如果没有则等待"""
        async with self._lock:
            await self._refill()
            while self.tokens < 1:
                # 计算需要等待的时间
                wait_time = (1 - self.tokens) / self.rate
                await asyncio.sleep(wait_time)
                await self._refill()
            self.tokens -= 1

    async def _refill(self):
        """补充令牌"""
        now = time.time()
        elapsed = now - self.last_refill
        new_tokens = elapsed * self.rate
        self.tokens = min(self.capacity, self.tokens + new_tokens)
        self.last_refill = now

    @property
    def available(self) -> float:
        """当前可用的 token 数"""
        return self.tokens


# 使用示例
async def main():
    bucket = AsyncTokenBucket(rate=10, capacity=20)  # 每秒 10 个，最多 20 个

    async def task(i):
        await bucket.acquire()
        print(f"Task {i} acquired token at {time.time():.2f}")

    # 同时启动 30 个任务
    await asyncio.gather(*[task(i) for i in range(30)])


if __name__ == "__main__":
    asyncio.run(main())
```
