## 一、项目定位与技术选型

### 1.1 解决什么问题

学术论文写作中存在大量**重复性劳动**：文献检索、数学建模、实验代码编写与执行、LaTeX 排版、同行评议。本系统将这些环节自动化。

### 1.2 为什么选 LangGraph 而不是 LangChain Agent

| 对比维度 | LangChain Agent | LangGraph |
|---------|----------------|-----------|
| 状态管理 | 隐式，依赖内存 | 显式 StateGraph + TypedDict |
| 流程控制 | 单一 while 循环 | 条件边 + 循环 + 并行 |
| 断点续传 | 不支持 | 原生 checkpoint |
| 可调试性 | 黑盒 | 每个节点独立，可单独测试 |
| 适用场景 | 简单工具调用 | 复杂多步工作流 |

**核心决策**：本系统需要 15+ 个 Agent 协作、条件分支、自动迭代、断点续传，LangGraph 是唯一合理选择。

### 1.3 技术栈全景

```
前端：Next.js 14 + React + Zustand + Tailwind CSS
后端：Python 3.9+ + FastAPI + LangGraph + asyncio
LLM：多 Provider（OpenAI/Anthropic/Kimi/DeepSeek）+ 5 种 API 格式
执行：Python 沙箱（namespace 隔离）+ GPUExecutor
存储：文件 JSON + SQLite（任务状态）
知识库：sentence-transformers + BM25 + RRF 融合
```

## 二、多智能体架构设计

### 2.1 Agent 抽象（BaseAgent）

所有 Agent 继承自 `BaseAgent`，统一接口：

```python
class BaseAgent:
    name: str           # 唯一标识
    label: str          # 中文标签
    description: str    # 功能描述
    default_model: str  # 默认 LLM 模型

    async def execute(self, task_input: Dict, context: Dict) -> Dict:
        """核心执行方法，子类必须实现"""
        pass

    async def call_llm(self, messages: List[Dict], **kwargs) -> Dict:
        """统一 LLM 调用（含重试、缓存、限流）"""
        pass

    def _inject_knowledge_context(self, context: Dict) -> str:
        """自动注入知识库内容"""
        pass
```

**设计要点**：
- `execute()` 返回 `Dict`，不是自定义类 → 方便序列化和跨 Agent 传递
- `call_llm()` 封装了所有 LLM 调用细节（重试、缓存、错误处理）
- 知识库注入是自动的，Agent 不需要手动查询

### 2.2 Agent 注册工厂

```python
@AgentFactory.register("solver_agent")
class SolverAgent(BaseAgent):
    name = "solver_agent"
    label = "求解器"
    # ...

# 使用时
agent = AgentFactory.create("solver_agent")
```

**为什么用工厂模式**：
- 解耦：Agent 定义和使用分离
- 可扩展：新增 Agent 只需加 `@register` 装饰器
- 可测试：测试时可以 mock 特定 Agent

### 2.4 输出归一化

三种建模 Agent（modeler/algorithm_engineer/financial_analyst）的输出统一为 `sub_problem_models`：


### 4.2 重试与限流

```python
# 令牌桶限流
class AsyncTokenBucket:
    def __init__(self, rate: float, capacity: int):
        self.rate = rate          # 每秒产生 token 数
        self.capacity = capacity  # 桶容量
        self.tokens = capacity
        self.last_refill = time.time()

    async def acquire(self):
        while self.tokens < 1:
            await asyncio.sleep(0.1)
        self.tokens -= 1

# 集成到 LLM 调用
async def call_llm(self, messages, **kwargs):
    await self.rate_limiter.acquire()  # 等待令牌
    try:
        return await self.client.call(messages, **kwargs)
    except RateLimitError:
        await asyncio.sleep(2 ** self.retry_count)
        return await self.call_llm(messages, **kwargs)  # 重试
```

### 4.3 LLM 调用缓存

```python
class LLMCache:
    """基于 SHA256 的请求级缓存"""

    def _hash_key(self, messages, model, **kwargs) -> str:
        content = json.dumps({"messages": messages, "model": model, **kwargs})
        return hashlib.sha256(content.encode()).hexdigest()

    async def get_or_call(self, messages, model, **kwargs):
        key = self._hash_key(messages, model, **kwargs)
        if key in self.cache:
            return self.cache[key]  # TTL 30 分钟
        result = await self.llm_client.call(messages, model, **kwargs)
        self.cache[key] = result
        return result
```

## 五、ReAct 工具循环

### 5.1 ReAct 原理

ReAct = **Re**asoning + **Act**ing，交替进行思考和行动：

```
Thought: 需要搜索关于 X 的论文
Action: web_search("X topic")
Observation: 找到 5 篇相关论文...
Thought: 还需要检查 arXiv
Action: arxiv_search("X")
Observation: 找到 3 篇论文...
... (重复直到完成)
```

### 5.2 动态迭代次数

```python
def _estimate_max_iterations(self, tools: List, context_length: int) -> int:
    """基于工具数量和上下文长度预估迭代次数"""
    base = len(tools) * 2  # 每个工具平均调用 2 次
    context_factor = context_length / 8000  # 上下文越长，需要越多迭代
    estimated = int(base * context_factor * 1.5)
    return min(max(estimated, 5), 20)  # 范围 [5, 20]
```

### 5.3 上下文压缩

```python
def _compress_react_history(self, history: List[Dict]) -> List[Dict]:
    """滑动窗口 + 摘要混合"""
    if len(history) <= 6:
        return history  # 最近 6 轮保留完整

    recent = history[-6:]  # 保留最近 6 轮
    old = history[:-6]

    # 旧的 tool call 压缩为摘要
    summary = self._summarize_old_steps(old)
    return [{"role": "system", "content": summary}] + recent
```

### 6.2 AST 代码审计详解

```python
# 检测规则
HARDCODED_METRICS = {"accuracy", "precision", "recall", "f1", "auc", ...}

class _MetricAssignDetector(ast.NodeVisitor):
    def visit_Assign(self, node):
        for target in node.targets:
            if isinstance(target, ast.Name):
                name = target.id.lower()
                if name in HARDCODED_METRICS:
                    if isinstance(node.value, (ast.Constant, ast.Num)):
                        # 发现硬编码指标赋值
                        self.issues.append(AuditIssue(
                            severity="error",
                            category="hardcoded_metric",
                            message=f"硬编码指标: {target.id} = {node.value.value}"
                        ))
```

**为什么用 AST 而不是正则**：
- 正则无法处理 `accuracy = some_function()` vs `accuracy = 0.95` 的区别
- AST 能精确识别赋值右侧是否为常量
- AST 能处理多行、复杂表达式

## 七、代码沙箱与安全

### 7.1 四层防御

```python
class CodeSandbox:
    def execute_file(self, path: str) -> ExecutionResult:
        # Layer 1: 静态分析
        self._static_analysis(path)

        # Layer 2: 资源限制
        self._set_resource_limits()

        # Layer 3: 网络隔离
        self._network_isolation()

        # Layer 4: 执行
        return self._run_in_subprocess(path)

    def _static_analysis(self, path: str):
        """正则扫描危险调用"""
        DANGEROUS_PATTERNS = [
            r"os\.system\(",
            r"subprocess\.call\(",
            r"eval\(",
            r"exec\(",
            r"__import__\(",
        ]
        code = open(path).read()
        for pattern in DANGEROUS_PATTERNS:
            if re.search(pattern, code):
                raise SecurityError(f"检测到危险调用: {pattern}")

    def _set_resource_limits(self):
        """POSIX 资源限制"""
        import resource
        resource.setrlimit(resource.RLIMIT_CPU, (120, 120))  # 120 秒 CPU
        resource.setrlimit(resource.RLIMIT_AS, (1024*1024*1024, 1024*1024*1024))  # 1GB 内存

    def _network_isolation(self):
        """网络命名空间隔离"""
        if os.path.exists("/usr/bin/unshare"):
            os.execvp("unshare", ["unshare", "--net", "python3", path])
        else:
            logger.warning("unshare 不可用，网络隔离降级")
```

## 八、知识库 RAG 系统

### 8.1 混合检索架构

```python
class HybridSearchEngine:
    def search(self, query: str, top_k: int = 5) -> List[Dict]:
        # 1. 语义检索
        semantic_results = self._semantic_search(query, top_k * 2)

        # 2. BM25 关键词检索
        bm25_results = self._bm25_search(query, top_k * 2)

        # 3. RRF 融合
        fused = self._rrf_fusion(semantic_results, bm25_results)

        return fused[:top_k]

    def _rrf_fusion(self, results_a, results_b, k=60):
        """Reciprocal Rank Fusion"""
        scores = {}
        for rank, doc in enumerate(results_a):
            doc_id = doc["id"]
            scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank + 1)
        for rank, doc in enumerate(results_b):
            doc_id = doc["id"]
            scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank + 1)
        return sorted(scores.items(), key=lambda x: x[1], reverse=True)
```

## 十、GPU 训练与实验执行

### 10.1 GPUExecutor

```python
class GPUExecutor:
    def execute_training(self, script_path: str, args: Dict, timeout: int) -> ExecutionResult:
        # 1. 检测 GPU
        if not torch.cuda.is_available():
            return self._run_cpu(script_path, args, timeout)

        # 2. 显存估算
        recommended_batch_size = self._estimate_batch_size(script_path)

        # 3. 启动训练
        process = self._start_training_process(script_path, args)

        # 4. 显存监控
        monitor = VRAMMonitor(process.pid)
        monitor.start()

        # 5. 等待完成
        try:
            stdout, stderr = process.communicate(timeout=timeout)
            return ExecutionResult(success=True, stdout=stdout, stderr=stderr)
        except subprocess.TimeoutExpired:
            process.kill()
            return ExecutionResult(success=False, error="训练超时")
```

### 10.2 VRAMMonitor

```python
class VRAMMonitor:
    def __init__(self, pid: int):
        self.pid = pid
        self.interval = 2  # 2 秒检查一次
        self.warning_threshold = 0.85  # 85% 预警
        self.kill_threshold = 0.95     # 95% 杀进程

    def _monitor_loop(self):
        while self.running:
            usage = torch.cuda.memory_allocated() / torch.cuda.get_device_properties(0).total_memory
            if usage > self.kill_threshold:
                logger.error(f"显存使用 {usage:.1%}，触发 OOM 保护")
                os.kill(self.pid, signal.SIGTERM)
            elif usage > self.warning_threshold:
                logger.warning(f"显存使用 {usage:.1%}，接近 OOM")
            time.sleep(self.interval)
```

## 十三、防死亡螺旋架构（v8.2）

### 13.1 问题定义

当 Coder Agent 生成复杂训练脚本时，沙箱执行失败可能级联为"死亡螺旋"——反复失败浪费计算资源且阻塞流程。即使代码能运行，系统也可能陷入"模板瓶颈"——指标因代码生成过于受限而停滞。

### 13.2 三机制设计

#### 机制 1：组件化注入（Component Injection）

**模块**：`_node_coder_agent()` in `langgraph_orchestrator.py`

在 `restricted` 模式下，Coder Agent 只能生成：
- `nn.Module` 子类（网络架构组件）
- Loss 函数组件
- 训练循环组件

系统自动将这些组件注入到预置的 Base Template 中：

```python
# Coder 输出（restricted 模式）：
# COMPONENT: nn.Module
class MyModel(nn.Module): ...

# COMPONENT: loss
def my_loss(pred, target): ...

# 系统自动组装为完整训练脚本
```

**设计意图**：受限模式下限制 Coder 的自由度，强制使用预验证的组件模板，降低代码出错概率。

#### 机制 2：AST 安全壳（AST Safety Shell）

**模块**：`safety_shell.py` + `audit_and_patch()` in `code_audit.py`

`SafetyShellTransformer` 执行 AST 级代码变换：

1. **try-except 包裹**：捕获所有异常，防止沙箱崩溃
2. **cuda.empty_cache() 注入**：在每个 `torch.cuda.*` 调用后注入
3. **gc.collect() 注入**：在 finally 块中注入内存清理

```python
# 变换前
model = MyModel().cuda()
output = model(data)

# 变换后
try:
    model = MyModel().cuda()
    output = model(data)
    torch.cuda.empty_cache()  # 注入
except Exception as _safety_exc:
    traceback.print_exc()
finally:
    gc.collect()  # 注入
```

#### 机制 3：渐进式越狱熔断（Progressive Jailbreak Circuit Breaker）

**模块**：`_node_reviewer_reflection()` + `_route_after_reviewer()` in `langgraph_orchestrator.py`

Reviewer Agent 监控执行指标，动态调整执行模式：

| 状态 | 条件 | 动作 |
|------|------|------|
| 正常运行 | `error_count < 3` | 保持当前模式 |
| 死亡螺旋 | `error_count >= 3` | 降级为 restricted，熔断阈值=3 |
| 模板瓶颈 | 指标连续 2 次未提升 | 升级为 jailbreak，熔断阈值=1 |
| 最大重试 | restricted 模式下仍连续失败 | 进入论文生成（带降级标记） |

### 13.3 全模板覆盖

所有经过 `iterative_solver` 的模板都接入 AST 安全壳 + 沙箱错误统计：

```
iterative_solver → ast_audit → sandbox_execution → figure/writer（非 CCF-A）
iterative_solver → coder_agent → ast_audit → sandbox → reviewer → writer（CCF-A）
```

- **CCF-A 模板**：完整流程（组件化注入 + 安全壳 + 越狱熔断）
- **非 CCF-A 模板**：简化流程（安全壳 + 错误统计，无组件化注入和越狱）
- **research_survey**：无代码执行，直接进入 writer
